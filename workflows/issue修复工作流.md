# Issue 修复工作流

> 适用于：已有功能的 BUG 修复、单点问题处理、线上事故回溯。
> 不适用于：新功能 / 跨模块重构（走 [需求实现工作流](需求实现工作流.md)）。
>
> 前置阅读：`AGENTS.md`、相关模块的 `design/*.md`、涉及模块的"不变量清单"一节。

---

## 工作流的存在理由

本项目踩过的坑：

- AI 修 BUG 的典型失败模式是"打补丁让症状消失"——在出 BUG 的位置加一个 `if (特殊情况) { 特殊处理 }`，让当次复现看起来通过。
- 几轮下来，特殊分支越堆越多，原本的主逻辑被淹没在补丁里。最后修一个新 BUG 时发现：根本看不出"主逻辑应该是什么"，整块代码无法维护。
- 根因：**修 BUG 没有经过架构校验，只做了症状定位**。

这份工作流的目的是强制在修复前先问"违反了哪条架构不变量"，答不出来就不能动代码。**核心原则**：

1. **修 BUG 前必须先诊断"违反了哪条架构不变量"**。答不出来，说明要么根因没找对，要么 design doc 的不变量清单不全——两种情况都应该先停下来补 doc，而不是继续打补丁。
2. **区分"修 BUG"和"绕过架构"**。如果修复的本质是加特殊分支，八成是后者——应该回到方案 doc 重新审视数据流，而不是继续在错误的地方打补丁。
3. **BUG 修复的产物不是只有代码**。每次修复都要回填 design doc（补一条不变量 / 修正一段描述），否则同一个 BUG 会换马甲再来一次。
4. **"症状消失" ≠ "修复完成"**。要能答得出"为什么之前会出、以后怎么防住同类问题"才算闭环。

---

## 五个阶段总览

```
阶段一：确认问题       —— 稳定复现、明确期望 vs 实际。
阶段二：校验架构       —— 找出违反的不变量，定位架构缺陷类型。
阶段三：明确方案设计   —— 修什么 + doc 回填什么 + 特殊判断检测。
阶段四：修复           —— 代码 + doc 同步改动，每步汇报。
阶段五：验证           —— 原 BUG 不再复现 + 不变量成立 + 回归 + doc 同步。
```

阶段不可跳。特别是阶段二——跳过架构校验直奔修复是本工作流想要阻止的头号反模式。

---

## 阶段一：确认问题

**谁主导**：用户 + AI 协作
**输入**：BUG 描述、用户观察到的症状
**输出**：issue 描述（可以写在 GitHub issue / Linear / 本地 markdown，不强求载体）
**退出条件**：能稳定复现 + 期望行为明确

### issue 描述模板

```
## 复现步骤
1. ...
2. ...
3. ...

## 期望行为
...

## 实际行为
...

## 环境
- 分支 / commit:
- OS / 浏览器 / 依赖版本（仅列与 BUG 相关的）:

## 相关日志 / 截图 / 录屏
...

## 初步怀疑范围
（文件、模块、数据路径。越具体越好。没把握就写"未知"，不要猜）
```

### 规则

- **稳定复现是硬门槛**。偶发 BUG 先想办法变稳定（加日志、缩小触发条件），不要直接进阶段二。真的无法稳定复现的，先单独写"加日志"这个小 issue，拿到数据再回来。
- **期望行为不能写"应该是对的"**。要能说清一个具体可验证的状态。
- **不要在阶段一下结论**。"我看了一眼代码，肯定是 X 行的问题"——这是阶段二的事，不是阶段一的事。

### 反模式

- ❌ "偶发问题" 直接进下一阶段 —— 一定会猜着修
- ❌ 期望行为模糊 —— 修完也没法验证
- ❌ 跳过 issue 描述直接 AI + 用户口头对一遍就开修 —— 没有白纸黑字，几天后谁也记不清当初说的是什么

---

## 阶段二：校验架构

**谁主导**：AI 分析，用户 review
**输入**：阶段一的 issue、相关 design doc、相关代码
**输出**：根因分析报告
**退出条件**：明确回答三个必答问题

### 三个必答问题

#### Q1：这个 BUG 违反了哪条架构不变量？

回到相关 `design/*.md` 的"不变量清单"一节，找出被违反的那条。

- **能定位到某条不变量**：说明不变量定义了但代码没守住——这是实现 BUG，继续阶段三
- **找不到对应不变量**：说明 design doc 的不变量清单不全——**先补 doc 再修代码**。补 doc 本身可能引出更多需要重新审视的代码，不要怕麻烦；这是最便宜的防再犯手段。

> **关键心智**：如果这一步答不出来就开始修代码，意味着你只知道"哪里出问题"，不知道"哪条规则被违反"。没有规则约束的修复就是打补丁——这次让症状消失了，下次同类问题还会从另一个地方冒出来。

#### Q2：为什么现有架构允许这个 BUG 出现？

按缺陷类型分类：

- **单一写路径被破坏**：某份状态有两条或以上路径在写
- **事件路由错配**：事件按隐式上下文（"当前 streaming 的那个"）而不是 id 路由
- **生命周期没收敛**：cancel / timeout / 正常结束三条路径没共用同一套收尾函数
- **默认值陷阱**：字段缺省时被 spread 成 undefined（AGENTS.md 明确点名过的坑）
- **术语漂移**：同一个词在不同模块里含义不同
- **流式 / 进程收尾缺陷**：见 AGENTS.md "流式输出与进程收尾" 一节
- **派生缓存与 source state 不同步**：两份同构缓存互相打架
- **其他**：说明是什么类型

再问一句：是"实现时就错了"还是"后续修 BUG 时逐步走形的"？

- **实现时就错了**：本工作流的阶段三修它
- **修 BUG 走形的**：说明之前的修复跳过了本工作流。这次修的同时要回溯最近 N 次相关修改，看看还有没有别的"打补丁"留下的隐患

#### Q3：这个 BUG 是孤例还是模式？

- 在相同模块 / 相同不变量下，是否可能还有其他未暴露的同类问题？
- **孤例**：直接修
- **模式**：阶段三的方案设计要考虑"一次修掉整类问题"，不要一个一个慢慢打

### 根因报告格式

```
## 根因分析

### Q1: 违反的不变量
{ 引用 design doc 的不变量 id + 原文 }
或：{ 现有 doc 无此不变量，建议新增: ... }

### Q2: 允许 BUG 出现的架构缺陷
类型：{ 从上面的分类中选一个 }
具体说明：...

是哪一类出身：
- [ ] 实现时就错了
- [ ] 后续修 BUG 逐步走形（追溯: 哪些 commit）

### Q3: 是否为模式
- [ ] 孤例
- [ ] 可能存在同类问题（列出怀疑点 + 代码指针）

### 相关代码指针
- file:line  现场位置
- file:line  同类可疑点 1
- file:line  同类可疑点 2
```

用户必须逐个回答确认，才能进阶段三。

## 阶段三：明确方案设计

**谁主导**：AI 起草，用户拍板
**输入**：阶段二的根因报告
**输出**：修复方案（可以内联在 issue 描述里，简单 BUG 不需要单独文件；涉及重构时才升级为独立 doc）
**退出条件**：通过"特殊判断检测"

### 方案要包含的内容

1. **修什么**：具体的代码改动点（文件 + 函数 + 大致思路）
2. **doc 回填**：哪条不变量要新增 / 哪段描述要修正。**不能为空**——如果真没什么可回填的，说明这个 BUG 没有教训可提炼，多半是根因没挖够
3. **影响面**：这次修改会触及哪些文件，哪些现有功能需要回归测试
4. **不做什么**：阶段二 Q3 发现的同类可疑点，如果不在本次影响面内，明确记录成后续 issue，**不要顺手改**（AGENTS.md "不做无关重构"）

### 特殊判断检测（critical）

看一眼修复方案。如果代码改动的本质是：

```ts
if (某个特殊情况) {
  // 特殊处理
} else {
  // 原逻辑
}
```

**停下来，不要继续**。这通常意味着：

- 你在绕过架构而不是修复它
- 下一次遇到另一个特殊情况，会再加一个 `if`
- 几轮之后特殊分支就变成了主逻辑，架构彻底烂掉

**正确做法**：回到阶段二 Q1，多半是根因没找对——你以为在修"表面不变量"，但其实更深一层的数据流约束才是被违反的那条。要么找到真正的不变量，要么承认 design doc 需要新增一条更抽象的不变量。修代码前先把 doc 的不变量改对。

### 可以豁免特殊判断的情况

- **业务本身就有特殊规则**（例如"管理员用户不受限额约束"——这是业务事实，不是架构缺陷）
- **外部系统的边缘数据**（例如第三方 API 偶发返回空、网络分区下的重试幂等处理）

豁免必须在方案里**显式写出**："这是业务 / 外部事实，不是架构绕路，理由是 XXX"。写不出理由就不属于豁免。

### 方案格式

```
## 修复方案

### 修什么
- file:line  改动描述
- file:line  改动描述

### doc 回填
- design/{相关 doc}.md "不变量清单" 新增一条 Ixx: ...
- design/{相关 doc}.md "{相关章节}" 修正描述：...

### 影响面
- 触及文件：...
- 回归面：{ 列出需要冒烟验证的现有功能路径 }

### 不做什么（本次 out of scope）
- 同类可疑点 A：已开后续 issue #...
- 潜在重构点 B：记录，本次不动

### 特殊判断检测
- [ ] 本次修复是否引入新的 `if (特殊情况)` 分支？
  - 否 → 继续
  - 是 → 豁免理由：...（必须填写）
  - 是且无豁免 → 停下，回阶段二
```

## 阶段四：修复

**谁主导**：Codex / Claude 执行
**输入**：阶段三的方案
**输出**：代码改动 + doc 回填（同一次提交里）
**退出条件**：AI 汇报完成清单 + 用户确认

### 强制动作

1. **按方案改代码**
2. **同时更新 design doc**（补不变量 / 修描述）——不允许"先改代码，doc 下次再说"。doc 和代码必须在同一阶段落地
3. **发现方案遗漏时停下来回阶段三**，不要边改边扩大范围
4. **修复完成后汇报**：

   ```
   ## 修复汇报

   ### 代码 diff 摘要
   - file:line  改动内容

   ### doc diff 摘要
   - design/{相关 doc}.md  回填内容

   ### 是否引入新的特殊分支？
   - 否 / 是（豁免理由：...）

   ### 阶段二 Q3 列的同类可疑点
   - 点 A：本次未处理，已开 issue #...
   - 点 B：本次未处理，已开 issue #...

   ### 方案外的代码触碰
   - 无 / 有（说明并解释必要性）
   ```

### 违规信号

- AI 只改代码不改 doc
- AI 引入新的 `if (特殊情况)` 分支且没在阶段三豁免
- AI 顺手重构方案外的代码（违反 AGENTS.md "不做无关重构"）
- AI 汇报时绕开"是否引入特殊分支"这个问题
- AI 边修边改方案（方案本身要先改好再进本阶段）

---

## 阶段五：验证

**谁主导**：AI 自检，用户终审
**输入**：阶段四的修复
**输出**：验证报告
**退出条件**：下面四项全部通过

1. 原 BUG 按阶段一的复现步骤再跑一遍，不再出现
2. 新 / 已有相关不变量都成立
3. 回归面：相同模块的冒烟路径都过一遍，前端改动必须浏览器打开肉眼验证
4. doc 和代码一致（grep 禁用词 + 人工对照）

### 验证报告模板

```
## 修复验证

### 1. 原 BUG 复现
按阶段一复现步骤重跑 → 不再出现 ✓
（附录：截图 / 日志）

### 2. 不变量核对
- Ixx（新增 / 修改）: ... → 成立 ✓
  验证手段：{ 单测 / 类型 / grep / 人工 }
- Iyy（已有）: ... → 仍成立 ✓

### 3. 回归
- 冒烟路径 A：通过 ✓
- 冒烟路径 B：通过 ✓
- 前端肉眼验证：已打开浏览器确认 ✓

### 4. doc 同步
- design/{相关 doc}.md 已更新
- 禁用词 grep：无命中

### 5. 遗留
- 同类可疑点 issue：#...（阶段二 Q3 发现但未在本次修复）
- 已知限制：...
```

### 反模式

- ❌ "按新的复现步骤跑也没问题了" —— 必须按**原**复现步骤，不能偷偷改复现条件
- ❌ "类型检查过了，应该不会回归" —— 回归面该跑的还要跑
- ❌ "doc 明天补" —— 今天不补就等于永远不补

---

## 工作流之间的关系

- 阶段二 Q1 发现"这不是修 BUG 而是架构本身有缺陷" —— 升级到 [需求实现工作流](需求实现工作流.md)，重走方案设计
- 阶段三的特殊判断检测触发且无法豁免 —— 不要强行修，回需求实现工作流重新设计数据流
- 阶段五完成后，相关 design doc 的更新就是本次 issue 的一部分闭环动作；doc 没更新 = 修复没完成
