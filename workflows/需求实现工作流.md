# 需求实现工作流

> 适用于：从零实现一个还没做过的功能，或者在已有功能上加新的能力。
> 不适用于：
> - 修 BUG → 走 [issue 修复工作流](issue修复工作流.md)
> - 推翻既有模块重做架构 → 不走本工作流，本工作流不处理"旧代码已经烂了怎么办"的问题；应该按重构需求单独写方案 doc（可参考 `design/chat-runtime-改进方案.md` 的结构）
>
> 前置阅读：`AGENTS.md`、`design/DESIGN.md`、`design/INDEX.md`。本文档不重复那里已经写过的代码规范，只约束"从需求到实现"的流程节奏。

---

## 工作流的存在理由

给 AI 只丢一段需求描述就让它写代码，典型失败模式有三种：

1. **术语跟既有代码撞车**：AI 引入的新名词和老代码里已有的概念语义重叠但叫法不同，后续每次改动都要花心思分辨"这里的 message 是哪种 message"。
2. **范围不受控**：AI 顺手改了本不该动的地方，或者把简单需求实现成一个过度设计的小框架。
3. **不留存档**：功能做完了没有留下可追溯的设计决定，下一次有人在这个功能上修 BUG 时，等于从零理解一遍。

这份工作流的目的是在"需求"和"代码"之间加三层缓冲——PRD、方案设计、不变量——让新代码落地时已经有完整的约束和存档。**核心原则**：

1. **不从需求直奔代码**。任何非 trivial 的新功能都要先产出方案 doc，用户 review 通过后再动手。
2. **术语先锁死**。方案 doc 第一节就是术语表；新功能引入的概念必须显式命名，且不能和既有概念撞车。
3. **不变量比测试用例更重要**。"测试设计"的核心是列出"新代码上线后必须永远满足的断言"，测试用例是验证手段。
4. **实现分阶段，阶段间有人工 checkpoint**。不允许一口气把几百行铺完——早一步截停总比晚一步好。
5. **方案 doc 是交付物的一部分**。代码交付时同步留下方案 doc，下一次维护才有据可查。

---

## 五个阶段总览

```
阶段一：PRD              —— 说清"要做什么 / 不做什么 / 成功长什么样"。
阶段二：方案设计         —— 术语锁定 + 数据 / 事件 schema + 对接点 + 推进顺序。
阶段三：测试设计         —— 不变量清单 + 验证手段，回填方案 doc。
阶段四：功能开发         —— 分步实现，每步人工 checkpoint 放行。
阶段五：验证闭环         —— 功能验证 + 不变量核对 + 文档归档。
```

阶段不可跳、不可合并、不可并行。每个阶段退出条件未满足，下一阶段不得开始。

## 阶段一：PRD（产品需求文档）

**谁主导**：用户
**输出**：`design/prd/{feature-name}.md`（或在既有 design doc 里新增一节，不强求单独文件）
**退出条件**：下列问题全部答清楚

### 必答问题

1. **这个功能解决什么问题？谁会用？在什么场景下用？**
2. **用户可感知的行为是什么？**（描述他能看到 / 操作什么，**不描述怎么实现**）
3. **成功长什么样？**（一个具体可验证的终态，不是"体验更好"这类模糊描述）
4. **不做什么？**（显式列出范围外的事项。范围蔓延 90% 来自这一条没写）
5. **有哪些硬约束？**（性能、兼容性、依赖、截止时间。如果没有，写"无"）

### 规则

- PRD 不写技术方案。出现"用 X 技术实现 Y"就挪到阶段二，除非它是硬约束（并说明为什么必须是这个技术）。
- "不做什么" 这一条写不清楚，说明需求本身还没想清楚——**不要进阶段二**。模糊的 PRD 一定会生出跑偏的方案。
- 需求模糊时，用户应该自己先想清楚，不要让 AI 帮你拍板。

## 阶段二：方案设计

**谁主导**：AI 起草，用户逐节拍板
**输入**：阶段一的 PRD、`AGENTS.md`、相关 `design/*.md`
**输出**：`design/{feature-name}.md`
**退出条件**：下述每一节都有内容，用户逐节确认

### 强制结构

#### 0. 术语约定（最重要，必须优先写完）

- 列表形式定义新功能引入的所有关键概念：
  - 定义（一句话说死）
  - 出现在哪一层（数据模型 / 协议 / 前端类型 / UI 文案 / 持久化）
  - 和外部系统（OpenAI / 其他 API）对应字段（如果有）
- **防撞车检查**：对每个新术语，grep 一遍现有代码和 `design/` 文档，确认没有既有概念在用这个名字。如果撞了，二选一：
  - 改新概念的名字
  - 确认二者就是同一东西，复用既有命名，不在 doc 里重复定义
- **禁用词列表**（可选）：如果引入的概念容易和某些通用词混淆，明确禁止那些词出现在新代码里

> 为什么这节最重要：术语不锁死，后续每一次修 BUG 都会让概念漂移一点。锁死术语是整份方案 doc 里对未来维护成本影响最大的一节。

#### 1. 需求摘要

把 PRD 浓缩成 3-5 句话放在方案 doc 开头，让读这份 doc 的人（包括未来的 AI 和未来的你）不需要反复跳回 PRD。**不要在这里加设计内容**——只是摘要。

#### 2. 对接点梳理（feature 专属，代替"问题诊断"）

新功能不是凭空出现的，它要和现有代码接上。这一节说清楚：

- **读哪些既有数据**：要从哪些现有的 store / 数据库表 / 接口里读东西
- **写哪些既有数据**：会往哪些现有结构里写东西（**这是最危险的一项**——写路径变多往往破坏"单一写路径"不变量，要在这里点名哪条路径是新增的）
- **订阅哪些既有事件**：会消费哪些运行时事件
- **发射哪些既有事件**：会在既有事件流里注入新事件
- **哪些既有 UI 区域被本功能改动**：前端改动点
- **本功能完全新建、不 touch 既有代码的部分**：单独列出来（通常是新组件、新 store、新后端模块）

每一项给代码指针（文件 + 函数 / 类型）。

**不是**在这里分析"现有架构有什么毛病"——那是重构方案该做的事，本工作流不处理那种情况。如果在对接点梳理阶段发现既有架构根本容不下这个需求，停下来和用户确认是不是应该升级成重构方案。

#### 3. 目标模型

新功能涉及的数据结构、事件、接口，**字面写 TypeScript / Rust 伪代码**，不要用自然语言描述"大概有哪些字段"。

- 新增的类型定义（`type Foo = { ... }` 完整写出）
- 新增的事件 schema（字段名、类型、可选性全写出）
- 新增的接口签名（输入、输出、错误）
- 数据流向图（文字或 ASCII）：谁产生数据 → 经过谁处理 → 到达谁

> 规则：能用代码表达的东西不要用散文表达。散文留给 AI 的自由度太大。

#### 4. 关键交互

- 主要用户流程对应的事件时序（ASCII 流程图或按步骤列表）
- 并发 / 异常 / 边界情况各自的预期行为（超时、取消、输入为空、权限不足等）
- **被否的替代方案**：列 1-2 个考虑过但没选的方向 + 否掉的理由。不是凑字数——这一段最大的价值是防止未来回头争论"为什么不那样做"

#### 5. 改动计划

按"新增 / 调整"两类平铺（新功能场景下"删除"通常为空，有的话才写）：

- **新增**：新建的文件、类型、组件、事件、接口，对应的路径和职责
- **调整**：既有代码需要改的地方（通常来自阶段 2 的对接点梳理），对应到具体文件 + 函数 + 改什么
- **删除**（可选）：只在本次新功能替代了某些临时占位代码 / mock 时才有。没有就写"无"

每一项能对应到具体路径和函数名。**写不出来就意味着方案还不具体**。

#### 6. 推进顺序

- 按可独立验证的粒度切分步骤（不强求"每步可发布"，但至少每步要能"跑起来 / 看得到"）
- 每步列出：前置依赖、改动范围、退出信号（怎么算这一步做完了）
- 相邻两步之间插入人工 checkpoint（阶段四会用到）
- 步骤数超过 6-8 步意味着要么粒度太细要么方案太大，考虑拆子需求

推进顺序不是"代码行号顺序"，是"功能可见度顺序"——先让最小闭环能跑通，再往上叠细节，这样每一步都能让用户实际验证。

#### 7. 不变量清单

此节在阶段三产出后回填到本方案 doc。

#### 8. 与现有 design 文档的关系

- 本方案引用了哪些既有 `design/*.md`？
- 本方案是否需要在既有 design doc 里补一段或改一段？**如果需要，必须同步改**——AGENTS.md 明确要求设计讨论的落点是 `design/`，不要在代码里另起炉灶

### 方案 review 要点

用户 review 时至少问一遍：

- 术语表里有没有漏掉你心里想着但 doc 没提的概念？
- 数据结构的伪代码里有没有"以为 AI 会懂"的字段？
- 对接点梳理里是不是漏了哪些既有模块其实会被 touch？（漏掉的对接点是事后"功能落地以后又发现和 X 冲突"的主要来源）
- "不做什么"和阶段一 PRD 里的"不做什么"是否一致？方案阶段是否偷偷扩大了范围？
- 推进顺序里的每一步是不是都能独立验证？

## 阶段三：测试设计

**谁主导**：AI 起草，用户 review
**输入**：阶段二的方案 doc
**输出**：回填到方案 doc 的"7. 不变量清单"一节
**退出条件**：每条不变量都有对应的验证手段，用户确认

### 核心观念：不变量 ≠ 测试用例

- **不变量（Invariant）**：新功能上线后，代码必须永远满足的断言。例如"用户只能看到自己创建的 X"、"payload 里的字段 Y 在缺省时不会被覆盖成 undefined"、"每个 spawn 的任务都有对应的 cancel 路径"。
- **测试用例**：验证不变量的具体手段。一条不变量可以对应多条测试；一条测试如果不对应任何不变量，多半是垃圾测试。

测试设计的任务：先把不变量列全，再为每条不变量选**最便宜的**验证手段。

### 新功能场景下值得考虑的不变量类型

- **权限 / 可见性**：只有满足某条件的用户 / session 能看到 / 操作这个东西
- **幂等性**：同一个输入重复触发不会产生重复副作用
- **单一写路径**：这个新功能引入的新状态只能被某一个入口修改
- **字段稳定性**：payload 缺省时某字段不会被 spread 成 undefined（AGENTS.md 明确点名过的坑）
- **生命周期完整**：每个 spawn 的任务 / 打开的资源都有对应的收尾路径
- **顺序 / 单调**：某 id / seq / cursor 严格单调
- **对接点约束**：本功能对既有数据的写操作，不破坏既有模块原有的不变量（需要先确认既有模块的不变量是什么；如果 design doc 里没写，这是顺手补一条的机会）

### 每条不变量的记录格式

```
### Ixx: {一句话断言}

**违反后果**：{违反会导致什么用户可见症状或数据损坏}

**验证手段**（按便宜程度排序选一种）：
- 类型系统保证（编译期）—— 最便宜，优先
- 单测 / 集成测试断言（运行期）
- 运行时 debug assert + 日志（兜底，要说明为什么前两种不可行）
- 人工走查（最后兜底，要说明为什么无法自动化）

**验证测试用例**：{对应的测试文件 / 用例名}（阶段四完成后回填）
```

### 什么时候要写集成测试 / 端到端测试

- 涉及前后端协议、流式传输、多进程协作：必须有集成测试，单测覆盖不到协议层
- 涉及 UI 视觉行为（动画、交互反馈、滚动）：必须在阶段五人工肉眼验证（AGENTS.md 明确要求前端改动要在浏览器里走一遍）
- 纯数据处理、纯逻辑函数：单测够用

## 阶段四：功能开发

**谁主导**：Codex / Claude 按方案执行
**输入**：方案 doc（阶段二 + 三）
**输出**：按推进顺序分步提交的代码改动
**退出条件**：每一步通过人工 checkpoint，全部步骤完成

### 强制规则

1. **严格按推进顺序执行**，不合并步骤、不跳步、不"顺手把下一步也做了"
2. **每完成一步，AI 必须汇报**以下内容，用户确认后才能进下一步：

   ```
   ## 步骤 N 完成汇报

   ### 动了哪些文件
   { git status 输出 }

   ### 改了哪些函数 / 类型
   - file:line  函数名  改动类型（新增 / 修改 / 删除）

   ### 是否触碰到方案外的文件？
   { 是 / 否。是的话说明为什么，以及是否需要更新方案 doc }

   ### 是否引入了方案 doc 里没有的新概念 / 抽象？
   { 是 / 否。是的话**停下来先更新方案 doc 第 0 节**再继续 }

   ### 本步骤的退出信号达成了吗？
   { 对照方案里写的退出信号一一核对 }

   ### 下一步准备动什么？
   { 简述 }
   ```

3. **人工 checkpoint 未通过前，下一步不得开始**
4. **不做方案外的改动**。阅读代码时若发现值得重构的点（AGENTS.md "边实现边识别" 一节），如果不在本次功能影响面内，记录成后续 issue，**不要顺手改**
5. **术语守护**：新写的类型名、函数名、变量名都要去方案 doc 第 0 节对照，不允许出现 doc 里没有的新概念。AI 觉得需要引入新概念时必须先改 doc

### 违规信号（出现就叫停）

- AI 引入了方案 doc 里没提过的新概念、新类型、新抽象
- AI 在一个步骤里触碰超过方案声明的文件集
- AI 出现"顺便也优化了一下 X"这种范围蔓延的描述
- AI 汇报含糊（"大致完成了"、"应该没问题"）
- AI 开始加 `if (特殊情况) { 特殊处理 }` 这类补丁式分支——新功能不该出现这种结构，出现了说明需求或方案里漏掉了一种情况

叫停后：回到方案 doc，补齐遗漏（必要时新开一版方案），或者让 AI 撤回本步重来。

## 阶段五：验证闭环

**谁主导**：AI 先自检，用户终审
**输入**：阶段四完成的代码
**输出**：验收报告
**退出条件**：验收报告所有项通过，相关 design doc 已同步

### 验收报告模板

```
## {feature} 验收报告

### 1. 功能行为核对
按 PRD 的"成功长什么样"逐项验证：
- [x] 行为 A：...
- [x] 行为 B：...
- [x] "不做什么"列表里的事项确实没做

### 2. 不变量逐条核对
- [x] I1: ... → 验证手段 / 结果
- [x] I2: ... → 验证手段 / 结果
- [ ] I3: ... （未通过的说明原因和补救方案）

### 3. 对接点回归
阶段 2 "对接点梳理"里列出的每个既有模块，跑一遍冒烟路径：
- [x] 既有模块 A：未回归
- [x] 既有模块 B：未回归
前端改动必须浏览器打开肉眼验证（AGENTS.md 要求）

### 4. 术语一致性
grep 新功能相关的关键术语，确认代码里的用法和方案 doc 第 0 节一致：
- 术语 X：代码命中 N 处，全部一致
- 防撞车：grep 禁用词（如果有）无命中

### 5. 文档归档
- [x] design/{feature}.md 与最终实现一致（差异已回填）
- [x] 既有 design doc 需要同步的地方已同步（如有）
- [x] PRD 里的"不做什么"最终都没做

### 6. 遗留
- 后续优化点（已开 issue）：...
- 已知限制：...
```

### 反模式

- ❌ "测试都过了" → 不变量 ≠ 测试用例，要逐条核对
- ❌ "我肉眼看了一下没问题" → 按清单走
- ❌ "文档以后再补" → 方案 doc 归档是闭环动作之一，拖到下次就是永远

## 工作流之间的关系

- 本工作流结束后发现 BUG → 走 [issue 修复工作流](issue修复工作流.md)
- 阶段二对接点梳理时发现"既有架构容不下这个需求" → 本工作流暂停，升级成重构方案专项处理（参考 `design/chat-runtime-改进方案.md` 的结构独立写一份）
- 阶段四发现方案有重大遗漏 → 回阶段二补方案，不要边实现边改方案
- 所有阶段都以"更新 design doc"作为闭环动作之一。doc 不同步 = 阶段未完成
