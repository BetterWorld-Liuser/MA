# easysdd-feature-test-spec

本技能对应 `workflows/需求实现工作流.md` 的**阶段三:测试设计**。该工作流文档是权威源。

## 涉及的路径

本技能在正文里用自然语言术语引用路径(**PRD 文档**、**方案 doc**、**架构中心目录**),具体目录位置见主技能 `easysdd` 第二节"目录安排"——那是整个 easysdd 家族目录约定的唯一定义处。

> `{feature}` 是占位符,代表具体 feature 的**目录名**,格式是 `YYYY-MM-DD-{英文 slug}`(日期前缀用于排序,小写连字符 slug;详见主技能 `easysdd` 第二节"目录安排")。到本阶段,feature 目录已由 brainstorm 或 PRD 阶段创建好,这里只消费不新建。

## 你的职责

为这个功能起草不变量清单。**不变量不是测试用例**——这是本技能最容易被搞混的一点,务必内化:

- **不变量(Invariant)**:新功能上线后,代码必须永远满足的断言。例如"用户只能看到自己创建的 X"、"payload 缺省字段不会被覆盖成 undefined"、"每个 spawn 的任务都有对应的 cancel 路径"。
- **测试用例**:验证不变量的具体手段。一条不变量可以对应多条测试;一条测试如果不对应任何不变量,多半是垃圾测试。

你的任务:**先把不变量列全,再为每条选最便宜的验证手段**。

## 启动检查

1. **方案 doc 是否存在且完整**。检查方案 doc:
   - 第 0-6 节有内容(8 节里前 7 节)
   - 第 7 节是空占位(等你填)
   - 不存在 / 不完整 → 停下来,告诉用户先走 easysdd-feature-design 技能。
2. **读相关上下文**:
   - 方案 doc 全文(尤其第 2 节对接点、第 3 节目标模型、第 4 节关键交互)
   - PRD 文档
   - `AGENTS.md`(里面点名过一些常见坑)

## 不变量挖掘清单

逐类扫一遍,看每类在本功能里是否适用。**每类都要主动想一遍**,不要漏类。

### 1. 权限 / 可见性
- "只有满足某条件的用户/session 能看到/操作这个东西"
- 适用场景:任何引入"用户专属数据"的功能

### 2. 幂等性
- "同一个输入重复触发不会产生重复副作用"
- 适用场景:涉及写操作、消息发送、计费、外部 API 调用的功能

### 3. 单一写路径
- "本功能引入的新状态只能被某一个入口修改"
- 适用场景:任何引入新状态的功能。这个不变量是防止未来"东一榔头西一棒槌"乱改状态的根本保险

### 4. 字段稳定性
- "payload 缺省时某字段不会被 spread 成 undefined"
- 适用场景:涉及对象合并、partial update 的功能。AGENTS.md 明确点名过这个坑

### 5. 生命周期完整
- "每个 spawn 的任务/打开的资源都有对应的收尾路径"
- 适用场景:涉及定时器、订阅、子进程、长连接、文件句柄的功能

### 6. 顺序 / 单调
- "某 id / seq / cursor 严格单调"
- 适用场景:涉及流式数据、事件序列、增量同步的功能

### 7. 对接点约束
- "本功能对既有数据的写操作,不破坏既有模块原有的不变量"
- 适用场景:**只要方案第 2 节对接点里有'写既有数据',就必须列**
- 需要先确认既有模块的不变量是什么——如果既有架构 doc(架构中心目录下的 doc)里没写,这是顺手补一条的好机会

## 不变量记录格式

每条不变量按这个格式写:

```markdown
### I{编号}: {一句话断言}

**违反后果**:{违反会导致什么用户可见症状或数据损坏}

**验证手段**(按便宜程度选一种):
- [ ] 类型系统保证(编译期)—— 最便宜,优先
- [ ] 单测 / 集成测试断言(运行期)
- [ ] 运行时 debug assert + 日志(兜底,要说明为什么前两种不可行)
- [ ] 人工走查(最后兜底,要说明为什么无法自动化)

**验证测试用例**:{对应的测试文件 / 用例名}(阶段四完成后回填)
```

**编号规则**:I1, I2, I3...,在方案 doc 内全局唯一。

**便宜程度**优先级要严格遵守:能用类型系统就别用运行时,能用单测就别用人工走查。每往下一档都要说明"为什么上一档不可行"。

## 集成测试 / E2E 测试的硬性要求

不是所有不变量都能用单测覆盖。**下列情况必须明确写入验证手段**:

- **涉及前后端协议、流式传输、多进程协作** → 必须有集成测试,单测覆盖不到协议层
- **涉及 UI 视觉行为(动画、交互反馈、滚动)** → 必须在阶段五人工肉眼验证(AGENTS.md 明确要求前端改动要在浏览器里走一遍)
- **纯数据处理、纯逻辑函数** → 单测够用

挖完不变量后做一次扫描:有哪几条对应的功能涉及上述场景?如果有,明确在该不变量的"验证手段"里写"集成测试 + 人工浏览器验证"。

## 回填到方案 doc

写完所有不变量,用 Edit 工具把方案 doc 的第 7 节占位替换成完整内容:

```markdown
## 7. 不变量清单

### I1: ...
**违反后果**:...
**验证手段**:...
**验证测试用例**:(阶段四回填)

### I2: ...
...
```

**不要新建文件**。不变量必须和方案其他部分待在同一份 doc 里——拆开就再也没人会去看了。

## 退出条件

- [ ] 7 类不变量都主动扫过(适用就列,不适用就跳过)
- [ ] 每条不变量都有"违反后果"和"验证手段"
- [ ] 验证手段按便宜程度选,降级有理由
- [ ] 涉及协议/流式/多进程的不变量明确要求集成测试
- [ ] 涉及 UI 行为的不变量明确要求人工浏览器验证
- [ ] 已用 Edit 工具回填到方案 doc 第 7 节
- [ ] 用户确认不变量清单可以了

## 退出后

告诉用户:"不变量清单已回填到方案 doc 第 7 节,方案 doc 现已完整。下一步是阶段四:分步实现。可以触发 easysdd-feature-implement 技能。"

## 反模式(看到就停)

- ❌ 把不变量写成"应该测试 XX 函数返回正确结果" —— 这是测试用例,不是不变量
- ❌ 7 类不变量没扫,直接列了 2-3 条就交卷 —— 主动扫的目的是不漏类
- ❌ 所有不变量都用"单测验证" —— 偷懒,该用类型系统的没用
- ❌ 涉及前端 UI 的不变量没写"人工浏览器验证" —— AGENTS.md 硬要求
- ❌ 把不变量另起一个文件 —— 必须回填到方案 doc 第 7 节
- ❌ 用户没确认就宣布完成
