# easysdd-feature-implement

本技能对应 `workflows/需求实现工作流.md` 的**阶段四:功能开发**。该工作流文档是权威源。

## 涉及的路径

本技能在正文里用自然语言术语引用路径(**PRD 文档**、**方案 doc**),具体目录位置见主技能 `easysdd` 第二节"目录安排"——那是整个 easysdd 家族目录约定的唯一定义处。

> `{feature}` 是占位符,代表具体 feature 的**目录名**,格式是 `YYYY-MM-DD-{英文 slug}`(日期前缀用于排序,小写连字符 slug;详见主技能 `easysdd` 第二节"目录安排")。到本阶段,feature 目录已由 brainstorm 或 PRD 阶段创建好,这里只消费不新建。

## 你的职责

按方案 doc 的"推进顺序"一步一步实现这个功能。**不是冲到底,不是一口气铺完代码**。每一步完成后必须汇报、等用户放行才进下一步。这是工作流的硬约束,违反它会丧失工作流提供的早停保护,导致跑偏的代码累积到难以收拾。

## 启动检查

1. **方案 doc 完整性**。检查方案 doc:
   - 第 0 节(术语)有内容
   - 第 2 节(对接点)有具体代码指针
   - 第 3 节(目标模型)有伪代码
   - 第 5 节(改动计划)指到具体文件 + 函数
   - 第 6 节(推进顺序)切分步骤明确,有退出信号
   - **第 7 节(不变量清单)非空**
   - 任何一项不达标 → 停下来,告诉用户先走 easysdd-feature-design 或 easysdd-feature-test-spec 补齐
2. **读所有上下文**(动手前必读):
   - 方案 doc 全文
   - PRD
   - `AGENTS.md`
   - 第 2 节对接点里提到的所有既有代码文件(只读相关函数即可,不必通读)
3. **跟用户确认从哪一步开始**。通常是第 1 步,但也可能是接续上次中断。

## 强制规则

### 规则 1:严格按推进顺序

- 不合并步骤
- 不跳步
- **不"顺手把下一步也做了"**——这是最常见的违规

### 规则 2:每完成一步必须汇报

汇报使用下面的固定模板。**不允许"大致完成了"、"应该没问题"这种含糊汇报**。

```markdown
## 步骤 N 完成汇报

### 动了哪些文件
{运行 git status,贴出真实输出}

### 改了哪些函数 / 类型
- file:line  函数名  改动类型(新增 / 修改 / 删除)
- file:line  ...

### 是否触碰到方案外的文件?
{是 / 否。是的话说明为什么,以及是否需要更新方案 doc}

### 是否引入了方案 doc 第 0 节里没有的新概念 / 抽象?
{是 / 否。**是的话:停下来先更新方案 doc 第 0 节再继续**,不要先实现再补 doc}

### 本步骤的退出信号达成了吗?
{对照方案第 6 节里写的退出信号一一核对,逐条勾选}

### 涉及的不变量自检
{本步骤涉及方案第 7 节哪几条不变量?当前实现是否满足?如果有不变量需要测试用例支撑,有没有同步加上?}

### 下一步准备动什么?
{简述下一步范围}
```

汇报完后**停下来**等用户回复。

### 规则 3:人工 checkpoint 未通过前,下一步不得开始

用户回复"继续"或类似明确放行才进下一步。用户没回应 / 回应模糊,主动澄清。

### 规则 4:不做方案外的改动

阅读代码时若发现值得重构的点(AGENTS.md "边实现边识别" 一节),如果**不在本次功能影响面内**,记录成后续 issue,**不要顺手改**。

记录格式:

```markdown
> 顺手发现:{文件:行号} {问题简述}。不在本次范围,记录待后续 issue。
```

### 规则 5:术语守护

- 新写的类型名 / 函数名 / 变量名都要去方案 doc 第 0 节对照
- 不允许出现 doc 里没有的新概念
- 觉得需要引入新概念时,**先停下来改方案 doc 第 0 节,grep 防撞车,用户确认,再继续写代码**

## 违规信号(出现就主动叫停)

下面的任一信号出现,**你必须主动停下来告诉用户**,不要硬着头皮继续:

- 你引入了方案 doc 第 0 节没提的新概念 / 新类型 / 新抽象
- 你在一个步骤里触碰了超过方案声明的文件集
- 你写出了"顺便也优化了一下 X"这种描述
- 你的汇报内容含糊("大致完成了"、"应该没问题")
- 你开始加 `if (特殊情况) { 特殊处理 }` 这类补丁式分支——**新功能不应出现这种结构**,出现说明需求或方案漏掉了一种情况

叫停后的处理:

- 回到方案 doc,补齐遗漏(必要时新开一版方案)
- 或者撤回本步重来
- **绝不要"先这样吧,后面再改"**——这是技术债起点

## 不变量验证手段同步

第 7 节不变量里"验证测试用例"这一栏,在写测试时同步回填到方案 doc。**测试通过 ≠ 不变量验证完成**——要确认每条不变量都有具体的测试覆盖到。

如果某条不变量靠类型系统保证,在汇报里说明"该类型签名已落地,编译期保证 IX"。

## 退出条件

- [ ] 方案 doc 第 6 节的所有步骤都完成
- [ ] 每一步都做过 checkpoint 汇报且用户放行
- [ ] 没有违规信号未处理
- [ ] 第 7 节不变量的"验证测试用例"栏全部回填
- [ ] 没有"顺手发现"被偷偷修掉(都进了 issue 列表)
- [ ] 没有方案外的文件改动(或改动已同步更新方案 doc)

## 退出后

告诉用户:"所有步骤完成,方案 doc 已同步。下一步是阶段五:验证闭环。可以触发 easysdd-feature-acceptance 技能。"

**不要自己顺手开始写验收报告**——验收阶段需要独立的 checklist 节奏。

## 反模式(看到就停)

- ❌ 一口气把所有步骤都做完才汇报
- ❌ 汇报里写"修改了相关文件"而不列具体 file:line
- ❌ 看到方案外的代码可以改一下顺手就改了
- ❌ 引入新类型但没回去更新方案 doc 第 0 节
- ❌ 加 `if (用户是 X) { 特殊处理 }` 补丁分支而不停下来反思方案
- ❌ 用户没明确放行就进下一步
- ❌ 不变量第 7 节的"验证测试用例"栏始终是空的
