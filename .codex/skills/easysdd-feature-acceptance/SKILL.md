# easysdd-feature-acceptance

本技能对应 `workflows/需求实现工作流.md` 的**阶段五:验证闭环**。该工作流文档是权威源。

## 涉及的路径

本技能在正文里用自然语言术语引用路径(**PRD 文档**、**方案 doc**、**验收报告**、**feature 目录**、**架构中心目录**),具体目录位置见主技能 `easysdd` 第二节"目录安排"——那是整个 easysdd 家族目录约定的唯一定义处。

> `{feature}` 是占位符,代表具体 feature 的**目录名**,格式是 `YYYY-MM-DD-{英文 slug}`(日期前缀用于排序,小写连字符 slug;详见主技能 `easysdd` 第二节"目录安排")。到本阶段,feature 目录已由 brainstorm 或 PRD 阶段创建好,这里只消费不新建。

## 你的职责

为这个功能产出一份验收报告,**逐项核对**而不是"我看了一眼觉得没问题"。验收报告是工作流的闭环动作——没产出报告 = 工作流未完成。

## 启动检查

1. **代码已实现到位**。检查 git status / 最近提交,确认有本功能的代码改动。
2. **方案 doc 完整**。方案 doc 第 0-8 节都有内容,第 7 节不变量的"验证测试用例"栏已回填(由阶段四完成)。
3. **PRD 存在**。PRD 文档存在,作为验收"功能行为核对"的输入源。
4. **读全部上下文**:
   - PRD
   - 方案 doc(尤其第 2 节对接点、第 7 节不变量)
   - `AGENTS.md`
   - 本次功能的代码改动(用 git log / git diff 看)

## 验收报告模板

逐节填写,**不要跳节**。报告路径:验收报告(feature 目录已经存在——因为里面住着 prd.md / design.md,直接把报告落进去;目录位置见主技能 `easysdd` 第二节"目录安排")。

```markdown
# {功能名称} 验收报告

> 阶段:阶段五(验证闭环)
> 验收日期:YYYY-MM-DD
> 关联 PRD:{PRD 文档路径}
> 关联方案 doc:{方案 doc 路径}

## 1. 功能行为核对

按 PRD 第 3 节"成功长什么样"逐项验证:

- [ ] 行为 A:{描述 + 实测结果}
- [ ] 行为 B:{描述 + 实测结果}
- [ ] ...

按 PRD 第 4 节"不做什么"逐项核对:

- [ ] 范围外事项 X **确实没做**(grep / 代码 review 确认)
- [ ] 范围外事项 Y **确实没做**
- [ ] ...

## 2. 不变量逐条核对

对方案 doc 第 7 节每条不变量:

- [ ] **I1**: {断言}
  - 验证手段:{类型 / 单测 / 集成测试 / 人工}
  - 结果:{通过 / 未通过 + 原因 + 补救方案}
- [ ] **I2**: ...
- [ ] ...

## 3. 对接点回归

按方案 doc 第 2 节"对接点梳理"列的每个既有模块,跑一遍冒烟路径:

- [ ] 既有模块 A({文件}):未回归 / 跑了哪几条 case
- [ ] 既有模块 B:...

**前端改动必须浏览器肉眼验证**(AGENTS.md 硬要求):

- [ ] UI 区域 X:浏览器验证 OK / 截图链接
- [ ] 交互行为 Y:浏览器验证 OK

## 4. 术语一致性

grep 新功能相关的关键术语,确认代码里的用法和方案 doc 第 0 节一致:

- 术语 X:代码命中 N 处,全部一致 ✓
- 术语 Y:代码命中 N 处,全部一致 ✓
- **防撞车**:方案 doc 第 0 节列的禁用词(如有),grep 无命中 ✓

如果有不一致,**停下来回到代码里改正**,不要在报告里写"未一致,以后再说"。

## 5. 文档归档

- [ ] 方案 doc 与最终实现一致(实现过程中的偏差已回填到方案 doc)
- [ ] 项目级架构 doc 需要同步的地方已同步(方案 doc 第 8 节列的架构中心目录下的 doc)
- [ ] PRD 第 4 节"不做什么"最终都没做(已在第 1 节确认)

## 6. 遗留

- 后续优化点(已开 issue 或加入 issue 列表):{列表}
- 已知限制:{列表}
- 阶段四"顺手发现"列表:{列表}
```

## 核对节奏

逐节做,不要跳:

1. 先做第 1 节(功能行为)和第 3 节(对接点回归)——这两节最容易暴露"实现没对齐需求"
2. 再做第 2 节(不变量)——逐条对照,涉及类型系统保证的让 typecheck 跑一遍,涉及单测/集成测试的让测试跑一遍
3. 第 4 节(术语)——用 Grep 工具搜索,把命中数和位置写进报告
4. 第 5 节(文档归档)——这是闭环动作,**doc 不同步 = 阶段未完成**
5. 第 6 节(遗留)——把阶段四攒下的"顺手发现"和已知限制都记进来

## 退出条件

- [ ] 验收报告 6 节都填完
- [ ] 第 1 节"功能行为核对"全部勾选
- [ ] 第 2 节"不变量"全部勾选(未通过的有补救方案)
- [ ] 第 3 节"对接点回归"全部勾选,前端改动已浏览器验证
- [ ] 第 4 节"术语一致性"无遗漏
- [ ] 第 5 节"文档归档"全部完成
- [ ] 用户终审确认

## 退出后

告诉用户:"验收报告已就绪,easysdd-feature 工作流走完。后续如发现 BUG,走 issue 修复工作流(不再回到本工作流)。"

可以建议:

- 提交时把方案 doc + 验收报告作为同一次提交的一部分,代码交付要带文档
- 验收报告里"遗留"的后续优化点,真要做的话另开新一轮 easysdd-feature 流程,不要在现有 PR 里塞

## 反模式(看到就停)

- ❌ "测试都过了" → 不变量 ≠ 测试用例,要逐条核对第 7 节
- ❌ "我肉眼看了一下没问题" → 按清单走,逐项勾选
- ❌ "文档以后再补" → 方案 doc 归档是闭环动作之一,**拖到下次就是永远**
- ❌ 第 4 节术语 grep 发现不一致就在报告里写"已知差异",而不去改代码
- ❌ 第 3 节前端改动只 typecheck 没在浏览器跑过 → AGENTS.md 硬要求
- ❌ 报告写完没让用户终审就宣告"工作流完成"
