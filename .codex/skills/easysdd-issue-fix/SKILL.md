# easysdd-issue-fix

本技能对应 easysdd-issue 工作流的**阶段三:修复验证**。

## 涉及的路径

本技能在正文里用自然语言术语引用路径(**问题报告**、**根因分析**、**修复备注**、**issue 目录**),具体目录位置见主技能 `easysdd-issue` 第二节"目录安排"。

## 你的职责

按根因分析确认的方案**定点修复**,**不是重构,不是顺手优化**。修完之后验证,把修复记录写进 fix-note.md。这是 issue 工作流的收尾动作——没有验证闭环 = 工作流未完成。

## 启动检查

1. **根因分析存在且用户已确认方案**。读 issue 目录下的 `analysis.md`,确认第 5 节"修复方案"用户选定了哪个方案。
2. **读全上下文**:
   - 根因分析(`analysis.md`)全文
   - 问题报告(`report.md`)全文
   - 根因分析第 1 节定位到的所有代码文件
   - `AGENTS.md`
3. **确认修复起点**:告诉用户"我将按方案 X 修改 {文件列表},开始修复"——等用户确认才动手。

## 强制规则

### 规则 1:只改根因分析里声明的文件

- 修复范围来自 analysis.md 第 5 节"推荐方案"里的"影响面"
- 超出范围的文件——哪怕顺眼——**不动**
- 发现范围外值得改的地方:记一条"顺手发现",不改代码

记录格式:
```markdown
> 顺手发现:{文件:行号} {问题简述}。不在本次修复范围,可后续另开 issue。
```

### 规则 2:改动最小化

- 修复的改动应该只针对根因,**不引入新的抽象、新的接口、新的模式**
- 如果发现"要把这个改好,需要先重构 X"——**停下来**,和用户确认是否要在这个 issue 里做重构,还是拆成独立的工作

### 规则 3:每完成一处改动必须汇报

汇报使用下面固定模板,**不允许含糊汇报**:

```markdown
## 修复汇报

### 动了哪些文件
{git status 输出或文件列表}

### 具体改了什么
- `{文件}:{行号}` — {改动描述,原来是 X,现在是 Y}
- ...

### 是否触碰到分析范围外的文件?
{是 / 否。是的话说明为什么}

### 是否引入了分析中没有提到的新概念/新结构?
{是 / 否。是的话停下来更新 analysis.md 后再继续}

### 复现步骤走一遍
{按 report.md 第 2 节复现步骤走一遍,结果是否符合期望行为?}
```

汇报完后**停下来**等用户回复。

### 规则 4:不做范围外的改动

和 easysdd-feature-implement 一样的逻辑:修复 issue 时发现的顺手优化机会,记录成新 issue,不顺手改。

## 验证清单

修复代码改完后,在汇报里逐项核对:

- [ ] **复现步骤验证**:按 report.md 第 2 节的步骤走一遍,问题不再出现
- [ ] **期望行为验证**:report.md 第 3 节描述的"期望行为"现在确实发生
- [ ] **影响面回归**:analysis.md 第 4 节"潜在受害模块"——每个模块走一遍最基本的冒烟路径,确认没被修复引入新问题
- [ ] **前端改动浏览器验证**(如果涉及):AGENTS.md 硬要求,不能只靠 typecheck
- [ ] **相关测试通过**:如果有测试覆盖到修复区域,跑一遍

## 写 fix-note.md

验证通过后,在 issue 目录下创建 `fix-note.md`(位置见主技能 `easysdd-issue` 第二节),记录本次修复的完整闭环:

```markdown
# {问题简述} 修复记录

> 阶段:阶段三(修复验证)
> 修复日期:YYYY-MM-DD
> 关联根因分析:{analysis.md 路径}

## 1. 实际采用方案

{方案 A / B,或说明与 analysis.md 确认方案的偏差及原因}

## 2. 改动文件清单

- `{文件}:{行号范围}` — {改动说明}
- ...

## 3. 验证结果

- 复现步骤验证:{通过 ✓}
- 期望行为验证:{通过 ✓}
- 影响面回归:{通过 ✓ / 发现问题 X(处理方式)}

## 4. 遗留事项

- {顺手发现的后续 issue(如有)}
- {已知限制(如有)}
- 无
```

## 退出条件

- [ ] 所有改动文件已提交或列清单
- [ ] 验证清单全部勾选
- [ ] fix-note.md 已创建并填写完整
- [ ] 没有未处理的"顺手发现"(都进了后续 issue 列表)
- [ ] 没有范围外的改动(或已和用户确认)
- [ ] 用户明确确认修复完成

## 退出后

告诉用户:"issue 修复完成,工作流闭环。问题报告(report.md) + 根因分析(analysis.md) + 修复记录(fix-note.md)已存档。"

可以建议:
- 把 issue 目录下的文件和代码改动放在同一次提交里,方便日后追溯
- "顺手发现"的后续 issue,另开一轮 `easysdd-issue-report` 处理,不要在这个 PR 里塞

如果修复过程中发现问题实际上是功能缺失(不是 BUG),建议用户另开 `easysdd-feature` 工作流处理,不要在 issue 工作流里偷偷做新功能。

## 反模式(看到就停)

- ❌ 修完代码没走验证清单就宣告"修好了"
- ❌ 顺手改了根因分析范围外的代码
- ❌ 修复引入了新的抽象/接口但没停下来和用户确认
- ❌ fix-note.md 没有创建就宣告工作流完成
- ❌ 发现影响面回归有问题,但在汇报里写"轻微影响,可以忽略"——不行,要修到干净
- ❌ 前端有改动但没在浏览器跑,只 typecheck 就报通过
- ❌ 用户没明确说"修复完成"就结束工作流
