# easysdd-feature-brainstorm

本技能对应 easysdd-feature 工作流的**可选 Stage 0:brainstorm**。它存在的唯一理由是:**当用户只有一个模糊想法时,直接让他回答 PRD 的 5 问会问不出任何有价值的东西——因为问题本身还没被框架好**。

本技能不是"另一种写 PRD 的方式",也不是"PRD 的前半段"。它是一个**独立的发散阶段**,目标是把"我想做点 X 吧"磨到"我明确要做 X 的这个切面,因为 Y,不做 Z",然后**交给 Stage 1 PRD 去锁死**。

> 参考来源:工作风格借鉴自 compound-engineering 插件的 `/ce:brainstorm` skill。本技能做了 easysdd 化的改写,不是照搬。

---

## 涉及的路径

本技能在正文里用自然语言术语引用路径(**feature 目录**、**brainstorm note**、**架构中心目录**、**架构总入口**、**所有 feature 的 PRD**、**所有 feature 的方案 doc**),具体目录位置见主技能 `easysdd` 第二节"目录安排"——那是整个 easysdd 家族目录约定的唯一定义处。

> `{feature}` 是占位符,代表具体 feature 的**目录名**,格式是 `YYYY-MM-DD-{英文 slug}`(日期前缀 + 小写连字符 slug,详见主技能 `easysdd` 第二节"目录安排")。brainstorm note 就住在该 feature 目录下的 `brainstorm.md`,和后续的 prd/design/acceptance 聚合在一起。由于 brainstorm 开始时 feature slug 可能还没最终确定,做法是:和用户商定一个**临时 slug**先用着,日期前缀取**开 brainstorm 当天**;如果 Stage 1 PRD 把名字改了,只重命名 slug 部分、日期前缀保留原值,连同 feature 目录一起 rename。

## 一、你的职责(以及不是你的职责)

**是**:
- 做一个**思考伙伴**:主动挑战、主动提议、主动挖隐含假设
- 一次问一个好问题,而不是一次抛一堆表单
- 开口前先快速扫一遍仓库,避免问已经有答案的问题
- 在用户开始收敛前,先做一次产品压力测试("这是不是真问题?不做会怎样?")
- 给出 2-3 个具体的可选方向(至少一个"反直觉"选项),让用户先看到选项再拍板
- 把对话沉淀成一份 brainstorm note,作为 PRD 的前置材料

**不是**:
- ❌ 不是要替代 PRD——本阶段结束必须交给 `easysdd-feature-prd`
- ❌ 不是要在本阶段锁死"不做什么"——那是 PRD 阶段的硬约束,本阶段只是让范围**开始有轮廓**
- ❌ 不是要讨论技术实现——"用什么数据库/框架/算法"一律推到阶段二(方案设计),本阶段只谈"用户感知的是什么 + 为什么做它"
- ❌ 不是要产出可以直接进阶段二的方案——本阶段产物是**发散记录**,不是 spec doc

## 二、启动检查

启动时先做三件事,**任何一件不满足就不继续**:

1. **确认这是新功能的 brainstorm,不是 BUG / 重构 / 方案讨论**
   - BUG 修复:明确告诉用户 brainstorm 不适用,走 issue 修复工作流
   - 既有模块重写:告诉用户应该写独立的重构方案 doc,不走 easysdd-feature 工作流
   - 已经想清楚、只需要写 PRD:告诉用户直接触发 `easysdd-feature-prd`,不需要走 Stage 0

2. **确认用户**真的**有模糊度需要发散**
   - 如果用户描述已经很清楚(有角色、有场景、有可验证终态),明说:"你的想法已经够清楚了,Stage 0 对你是多余的,建议直接进 Stage 1 PRD。"然后停。
   - 判断原则:如果你现在已经能帮用户写出 PRD 的第 1/2/3 问的初稿,那就不需要 brainstorm
   - 宁愿误放过(让用户直接进 PRD)也不要误揽(逼用户做他已经不需要的发散)

3. **检查是否已有同主题的 brainstorm note 或 feature 目录**
   - Glob 所有 feature 目录下的 `brainstorm.md`、`prd.md` 看有没有近义文件(brainstorm note 现在住在 `features/{feature}/brainstorm.md`,和 prd/design/acceptance 同址)
   - 有的话先确认"接着上次改"还是"推翻重开";如果是已走到 PRD 之后的 feature,默认不要覆盖其 brainstorm note,而是让用户确认是否另起一个新 slug
   - 绝对不允许默默另起炉灶,也不允许默默覆盖别人已有的 feature 目录

## 三、核心工作风格(从 /ce:brainstorm 吸收)

这是本技能和其他 easysdd 子技能最大的区别。**违反任何一条都是反模式**。

### 风格 1:一次一个问题

- 每轮对话**只提一个问题**,等用户回答再出下一题
- 不允许"我问你 5 个问题,你一起答":这会让用户在某一题上草草应付,也让你拿不到足够专注的答案
- 每个问题问完后,先做一次"理解反述"("所以你的意思是……对吗?"),再出下一题

### 风格 2:优先用多选题,不要让用户自由作文

- 提问时**优先用 `AskUserQuestion` 出单选/多选题**,每题给 2-4 个具体、互斥、有区别度的选项
- 什么时候用单选:方向类选择、优先级、下一步、定性判断
- 什么时候用多选:目标集合、约束集合、非目标集合——这些"可以共存"的东西
- **多选题的选项本身就是你的思考**:给出好的选项比问出好的问题更重要。选项必须具体到用户能立刻判断,不能是"A 方案 / B 方案"这种空壳
- **降级规则**:如果环境没有 `AskUserQuestion`,降级为编号文本选项("请回复 1/2/3,或补充其他"),保持一次一个问题和预设选项的风格
- 反模式:把"请描述一下你的需求"这种开放题抛给用户——那是 PRD 要做的事,brainstorm 是要帮用户收敛,不是加重他的认知负担

### 风格 3:做思考伙伴,不做问卷机

- **主动提议,不只是提问**:当你识别出一个用户还没想到的角度,直接说出来("有没有考虑过……?我猜测这里会有个问题……")
- **主动挑战**:如果用户的问题框架本身可疑(比如把手段当目的),直接指出来并给出重新框架的候选
- **主动带方案空间**:不要等用户告诉你有哪些选项,你来提议 2-3 个具体方向
- 反模式:"那你觉得应该怎么做呢?"——这是偷懒,是把问题踢回给用户。你应该说:"我能想到三种方向,A / B / C,你怎么看?"

### 风格 4:开口前先扫一遍仓库

- 在第一次提问**之前**,做一次轻量扫描:
  - Read `AGENTS.md`(项目硬约束和已知坑)
  - Read 架构总入口(架构权威,要对齐核心理念)
  - Glob 所有 feature 目录下的 `brainstorm.md` / `prd.md` / `design.md`,看是否有相关文件(easysdd 家族的现存产物——同一 feature 的 brainstorm/prd/design/acceptance 都聚合在该 feature 目录下)
  - 顺便 Glob 一眼架构中心目录下的 doc(架构总入口、架构索引、子系统架构 doc)看有没有相关的架构讨论
  - Grep 用户描述里的关键词,看代码里是否已经有近义概念(防术语撞车,即使本阶段不锁术语)
- 扫描结果**明确在对话里汇报**:"我先看了下仓库,发现 X 已经存在 / Y 最近有相关设计 / Z 在 AGENTS.md 里有硬约束……在这些约束下,我对你的想法有这些初步判断……"
- 不允许"空口问问题"——那样会问到仓库里已经有答案的东西,浪费用户时间
- 反模式:跳过扫描,直接开问——基本保证你问出的第一个问题就是个低质问题

### 风格 5:先做产品压力测试,再谈方案

在开始提"具体做什么"之前,先问这些高层问题**至少一轮**,哪怕用户会觉得"啰嗦":

- 这是在解决真问题,还是手段被当成了目的?
- 不做会怎样?不做的话,用户会去干什么?
- 有没有一个更高价值的邻近问题,用几乎相同的成本可以一起解决?
- 有没有一个"做更少"的版本能抓到 80% 的价值?
- 在这个仓库现在的状态下,这是不是此刻最该做的事?

**不是走过场**。如果压力测试暴露出"其实用户应该做另一件事",**立刻停下来告诉用户**,让他重新决定要不要继续 brainstorm 当前这个方向。宁可把用户导向一个更值得做的方向,也不要陪他把一个弱需求磨成漂亮的 PRD。

### 风格 6:提议 2-3 个方向,不要只给一个"最佳答案"

当对话进入"要做什么"的讨论时,**主动提出 2-3 个具体的候选方向**,每个都包括:

- 1-2 句话的描述(用户可感知的行为,不是技术实现)
- 它的核心价值点
- 它的主要代价 / 风险
- 一个"什么情况下这个方向最合适"的判断句

**至少有一个候选是"反直觉"或"换角度"的**:反转("如果我们不做这个功能,而是反过来做……")、约束去除("如果 X 不是限制条件,我们会怎么做?")、类比("别的系统是怎么解这个问题的?")。

**先把所有候选呈现完,再给推荐**——不要"我推荐 A 因为 balabala,顺便说下还有 B/C"。这种顺序会把用户锚定到 A,失去真正对比的机会。

## 四、执行流程

按下面的顺序走,每一步都对应上面的某个风格规则。

### 步骤 0:仓库扫描(对应风格 4)

- Read `AGENTS.md`、架构总入口(项目级架构权威)
- Glob 所有 feature 目录下的 `brainstorm.md` / `prd.md` / `design.md`(easysdd 家族产物,同一 feature 的所有 spec 都在自己的 feature 目录里)
- 顺便 Glob 架构中心目录下的 doc(项目级子系统架构 doc)看有没有相关架构讨论
- Grep 用户描述中的关键词
- **搜索知识沉淀目录**:用 `python easysdd/tools/search-yaml.py --dir easysdd/learnings --query "{用户描述中的关键词}"` 搜索 learnings 目录,看有没有相关的 pitfall 或 knowledge 文档。找到的话在汇报里一并提出("上次做类似方向时踩过 X 坑,本次 brainstorm 需要考虑进去")
- 在对话里汇报一次初步发现,然后进入步骤 1

### 步骤 1:问题框架确认(对应风格 5)

- 第一个问题**一定是关于问题本身**:"你想解决的真问题是不是……?还是其实是……?"(给 2-3 个候选框架)
- 如果用户选了一个框架,继续;选了"其他"并写了自由文本,你要把它再转化成可选项问一遍
- 这一步允许反复 1-3 轮,直到问题框架稳住

### 步骤 2:产品压力测试(对应风格 5)

- 问至少 2 个压力测试问题(见风格 5 的清单),每次一题
- 如果测试暴露出用户应该做别的事,在这里停下来,告诉用户并让他决定是否继续

### 步骤 3:提议 2-3 个方向(对应风格 6)

- 在对话里列出完整的候选清单(**所有**候选描述完再给推荐)
- 至少包含一个反直觉候选
- 用 `AskUserQuestion` 让用户单选一个方向(选项的 label 就是候选名,description 写核心价值+代价)
- 如果用户选"其他",继续一轮讨论再出新一批候选

### 步骤 4:边界粗勾(为 PRD 预热,但不是 PRD)

选定方向后,问这些问题(仍然一次一个,优先多选):

- 用户可感知的核心行为是什么?(粗粒度即可,PRD 会细化)
- 哪些事情**明确不做**?(列 2-3 条就够,PRD 会扩)
- 最大的未知/风险是什么?
- 成功长什么样?(粗粒度)

**不要试图在这一步就产出 PRD 级别的答案**——PRD 阶段会有专门的 5 问。你要的只是"足够进 PRD 的轮廓"。

### 步骤 5:写 brainstorm note

落盘到 brainstorm note(路径和命名约定见主技能 `easysdd` 第二节"目录安排";brainstorm note 住在 `features/{feature}/brainstorm.md`,和后续 prd/design/acceptance 同址)。

**确定 feature 目录名**:feature 目录名的格式是 `YYYY-MM-DD-{英文 slug}`(详见主技能 `easysdd` 第二节"目录安排")。具体做法:
- **日期前缀**:用**今天**的日期,自己不要凭空想,**先从环境信息里查今天是几号**(比如系统上下文里的 `currentDate`),拿到后再拼进去
- **slug 部分**:和用户确认一个**英文小写连字符的临时 slug**——哪怕此时他还不确定 feature 最终叫什么也没关系,先用着,Stage 1 PRD 如果改名就只 rename slug 部分、保留原日期前缀
- **不允许**在未与用户确认 slug 的情况下默默创建目录,也不允许随便编一个日期

**如果该 feature 目录不存在,就创建它**,并把 brainstorm note 作为第一个文件写进去。如果该 feature 目录已经存在(说明之前有过相关 spec),参考"启动检查"第 3 条,不要默默覆盖。

### 步骤 6:退出与交接

- 对照退出条件逐项核对
- 明确告诉用户:"brainstorm 到此结束。下一步是 Stage 1 PRD,触发 `easysdd-feature-prd` 技能开始。"
- **绝对不要**自己接着写 PRD——阶段间的人工 checkpoint 是 easysdd 的硬约束

## 五、brainstorm note 模板

**路径**:brainstorm note(目录位置见主技能 `easysdd` 第二节"目录安排")

**内容**:

```markdown
# {功能名称} Brainstorm

> 阶段:Stage 0(可选前置·brainstorm)
> 状态:草稿 / 已确认
> 创建日期:YYYY-MM-DD
> 下一步:Stage 1 PRD

## 1. 问题框架
{用户最终确认的"真问题"是什么,一段话说清楚。
注明对话里被讨论但被否掉的其他框架,以及为什么被否。}

## 2. 产品压力测试结论
- **不做会怎样**:{回答}
- **更高价值的邻近问题**:{有 / 无,如有列出并说明为什么本次不收}
- **更小版本**:{是否存在一个"做更少抓 80%"的版本,以及是否采用}
- **时机判断**:{在当前仓库状态下,这是不是此刻最该做的}

## 3. 考虑过的方向
逐一列出对话中提议过的 2-3 个方向,每个包括:

### 方向 A:{名称}
- 描述:...
- 核心价值:...
- 主要代价 / 风险:...
- 结论:{选定 / 否决,以及原因}

### 方向 B:...
### 方向 C:...

## 4. 选定方向
{选定的那一个方向,用 2-3 句话重新陈述,作为 PRD 的起点}

## 5. 粗粒度边界(供 PRD 参考,不是最终范围)
- **核心用户感知行为**:...
- **明确不做**(候选):
  - ...
- **最大未知 / 风险**:...
- **成功的粗粒度描述**:...

## 6. 遗留给 PRD 阶段的问题
{对话中识别出但本阶段不回答的问题,留给 Stage 1 PRD 继续挖}
- ...

## 备注
{可选:仓库扫描发现的相关文档/代码引用、术语撞车风险提示等}
```

## 六、退出条件

写完 note 后,跟用户做一次"出 Stage 0"确认:

- [ ] 问题框架已稳(用户能用一句话复述"真问题是什么",且和自己最初的描述有所不同或更精炼)
- [ ] 做过产品压力测试且结论明确(没有留"这个问题我们还没想")
- [ ] 至少呈现过 2 个方向供对比(不一定要 3 个,但不能只有 1 个"钦定方案")
- [ ] 粗粒度边界里"明确不做"列出至少 2 条
- [ ] brainstorm note 已落盘到对应的 feature 目录(`features/{feature}/brainstorm.md`)
- [ ] 用户明确说"可以进 PRD 了"

**任何一项不达标都不要交接到 Stage 1**。一份不彻底的 brainstorm 比没做 brainstorm 更糟——因为它会制造"我们已经想清楚了"的假象。

---

## 七、退出后

告诉用户:

> brainstorm note 已落盘在该 feature 的目录下(`features/{feature}/brainstorm.md`),和后续的 prd/design/acceptance 将聚合在同一个目录。下一步是 Stage 1 PRD:触发 `easysdd-feature-prd` 技能,我会基于这份 note 引导你回答 5 个必答问题,把范围锁死。如果在 PRD 阶段决定改 feature 名,整个 feature 目录一起重命名即可。

**不要自己顺手开始写 PRD**。brainstorm 和 PRD 之间的 checkpoint 是本工作流的硬约束——哪怕只隔 5 分钟,也要让用户显式发起下一阶段。

---

## 八、反模式(看到就停)

- ❌ **一次抛 5 个问题的清单**:这是 PRD 风格,不是 brainstorm 风格。brainstorm 必须一次一题
- ❌ **提问前不扫仓库**:问出"这个项目有没有 X"——而代码里就能搜到,浪费用户时间
- ❌ **只给一个"推荐方案",不给对比**:用户没看到候选就拍板,等于你替他做了决定
- ❌ **只让用户回答,不做主动挑战**:brainstorm 的价值一半在挑战上,只记录不挑战就是问卷
- ❌ **在本阶段讨论技术实现**:"要不要用 X 库"、"表结构怎么设计"——一律推到阶段二
- ❌ **在本阶段锁死"不做什么"**:那是 Stage 1 PRD 的第 4 问,本阶段只画轮廓
- ❌ **用户说"差不多就这样吧"就直接落盘**:必须严格走退出条件逐项核对
- ❌ **落盘后顺手开始写 PRD**:违反 easysdd 的阶段 checkpoint 硬约束
