# easysdd-feature-prd

本技能对应 `workflows/需求实现工作流.md` 的**阶段一:PRD**。该工作流文档是权威源,本技能只承载操作手册。如果你对某个规则的"为什么"有疑问,去读那份文档。

## 涉及的路径

本技能在正文里用自然语言术语引用路径(**PRD 文档**、**feature 目录**、**feature 聚合根**、**架构中心目录**),具体目录位置见主技能 `easysdd` 第二节"目录安排"——那是整个 easysdd 家族目录约定的唯一定义处。

> `{feature}` 是占位符,代表具体 feature 的**目录名**,格式是 `YYYY-MM-DD-{英文 slug}`(日期前缀用于排序,小写连字符 slug;详见主技能 `easysdd` 第二节"目录安排")。

## 你的职责

引导用户产出一份合格的 PRD,**不是替用户拍板需求**。PRD 的每一个回答都应该来自用户本人;你的职责是问对问题、识别模糊回答、防止用户提前跳进技术方案。

## 启动检查

启动时先确认三件事:

1. **用户给的是新功能,而不是 BUG 或重构**。
   - 如果是 BUG:告诉用户应该走 issue 修复工作流,本技能不适用。
   - 如果是"重写既有模块":告诉用户应该独立写重构方案 doc,参考架构中心目录下的 `chat-runtime-改进方案.md` 的结构,不要走本工作流。
2. **是否已经存在同名 PRD**。检查 feature 聚合根下有没有同名或近义的 feature 目录含 `prd.md`,有的话先和用户确认是新建还是接着改。
3. **是否存在 brainstorm note**。检查当前 feature 目录下有没有 `brainstorm.md`。有的话走"brainstorm → PRD 衔接模式"(见下节),没有就正常逐题问。

## brainstorm → PRD 衔接模式

**当 brainstorm note 存在时,PRD 不从零问起**。执行以下步骤:

1. **读 brainstorm note**,把已有内容映射到 5 问:
   - brainstorm §1"问题框架" → PRD 第 1 问(问题与场景)
   - brainstorm §4"选定方向" + §5"核心用户感知行为" → PRD 第 2 问(用户可感知行为)
   - brainstorm §5"成功的粗粒度描述" → PRD 第 3 问(成功长什么样)
   - brainstorm §5"明确不做" → PRD 第 4 问(不做什么)
   - brainstorm note 里提到的约束/风险 → PRD 第 5 问(硬约束)

2. **向用户呈现预填稿**:把从 brainstorm 里提取的内容逐问列出,明确告知哪些是从 brainstorm 继承的、哪些还需要补充或精确化。**不要让用户重新回答已经清楚的问题**——直接说"这一问我从 brainstorm 里提取了如下内容,请确认或补充"。

3. **只追问真正缺失或不够精确的部分**:brainstorm 的边界是粗粒度的,PRD 需要锁死。重点检查:
   - 第 2 问:brainstorm 里的"行为"描述是否混入了技术实现?有的话拦下来
   - 第 3 问:是否具体可验证?粗粒度描述需要追问到"怎么测才算达成"
   - 第 4 问:"明确不做"是否至少有 2-3 条?brainstorm 里通常只有候选,需要用户显式确认哪些是真正的范围外
   - 第 5 问:硬约束是否有遗漏?brainstorm 阶段没有专门问过约束

4. **brainstorm note 里第 6 节"遗留给 PRD 阶段的问题"**:如果 brainstorm 里明确列了遗留问题,PRD 必须把这些问题逐一处理,不能跳过。

> **原则**:PRD 是精确化和锁死,不是重新发散。brainstorm 已经充分讨论过的部分,PRD 阶段做确认+细化;brainstorm 遗留的缺口,PRD 阶段补上。

## 必答 5 问

按顺序逐个处理,**不要一次抛 5 个**。每个问题用户回答(或确认)完,你做一次"模糊度检查",不通过就继续追问。有 brainstorm note 时按衔接模式处理(见上节),以下是每问的精确化标准:

### 1. 这个功能解决什么问题?谁会用?在什么场景下用?

- 期待的回答形式:一段具体的场景描述,有明确的角色和触发情境。
- 模糊信号:"为了提升用户体验"、"让 XX 更好用"——追问到具体场景为止。

### 2. 用户可感知的行为是什么?

- 期待:用户能看到/操作什么,用"用户做 X → 看到 Y"的形式描述。
- 红线:**不允许出现技术实现描述**。出现"用 WebSocket 推送"、"加一个新表"——拦下来,告诉用户这属于阶段二,本阶段不写。

### 3. 成功长什么样?

- 期待:一个具体可验证的终态。
- 模糊信号:"体验更好"、"更稳定"、"更快"——追问"具体怎么测才算达成"。

### 4. 不做什么?(范围外)

- **这是最关键的一问**。范围蔓延 90% 来自这一条没写清楚。
- 如果用户说"我想到的都要做"——拦下来,逼他显式列出至少 3 条范围外事项。一条都列不出来通常意味着需求本身没想清楚,不要进阶段二。
- 范例追问:"这个功能要不要支持移动端?"、"要不要带权限控制?"、"老数据要不要迁移?"——挑几条用户没提的边界来反问,看他怎么答。

### 5. 有哪些硬约束?

- 性能、兼容性、依赖、截止时间、合规要求。
- 没有就明确写"无"。**不要留空**——空着等于没问过。

## PRD 模板

让用户回答完后,把回答写进文件:

**路径**:PRD 文档(目录位置见主技能 `easysdd` 第二节"目录安排")。

**确定 feature 目录名**:feature 目录名的格式是 `YYYY-MM-DD-{英文 slug}`。分两种情况:
- **如果 Stage 0 brainstorm 已经创建过 feature 目录**:默认**沿用原目录名**,不要改日期前缀;只有在用户明确说要改 slug 时才 rename 后半段 slug、保留原日期前缀
- **如果是直接从 PRD 启动(没有 brainstorm)**:和用户确认一个 slug,日期前缀用**今天**的日期——自己不要凭空想,**先从环境信息里查今天是几号**(比如系统上下文里的 `currentDate`),拿到后再拼完整目录名。feature 目录不存在就创建——同一 feature 的所有 spec 产物都聚合在这个目录下,不再散落到并列目录

**内容**:

```markdown
# {功能名称} PRD

> 阶段:阶段一(PRD)
> 状态:草稿 / 已确认
> 创建日期:YYYY-MM-DD

## 1. 问题与场景
{第 1 问的回答}

## 2. 用户可感知行为
{第 2 问的回答,纯行为描述,无技术细节}

## 3. 成功长什么样
{第 3 问的回答,可验证的终态}

## 4. 不做什么
- {范围外事项 1}
- {范围外事项 2}
- {范围外事项 3}
- ...

## 5. 硬约束
{第 5 问的回答,没有就写"无"}

## 备注
{可选:用户额外补充的上下文}
```

## 退出条件

写完文件后,跟用户做一次"出阶段一"确认:

- [ ] 5 个问题都有具体答案,没有"待定"
- [ ] 第 2 问没有混入任何技术实现
- [ ] 第 4 问"不做什么"列出至少 2-3 条
- [ ] 用户明确说"PRD 可以了,进下一步"

**任何一项不达标都不要进阶段二**。模糊的 PRD 一定生出跑偏的方案——这不是你的判断,这是工作流文档明文写的规则。

## 退出后

告诉用户:"PRD 已就绪,下一步是阶段二:方案设计。可以触发 easysdd-feature-design 技能开始。"

不要自己顺手开始写方案——阶段之间的人工 checkpoint 是工作流的硬约束。

## 反模式(看到就停)

- ❌ 用户回答"我想要一个 AI 推荐系统"就开始追问"用什么模型" —— 错,那是阶段二。
- ❌ 用户的"不做什么"是空的或"暂时没想到",你帮他编了几条 —— 错,这是用户必须自己想清楚的。
- ❌ 一口气把 5 个问题列成清单丢给用户填 —— 错,要逐题对话,捕捉模糊回答。
- ❌ 用户说"差不多了"就放行 —— 错,严格走退出条件。
