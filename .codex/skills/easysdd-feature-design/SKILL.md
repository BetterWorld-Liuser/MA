# easysdd-feature-design

本技能对应 `workflows/需求实现工作流.md` 的**阶段二:方案设计**。该工作流文档是权威源。

## 涉及的路径

本技能在正文里用自然语言术语引用路径(**PRD 文档**、**方案 doc**、**feature 目录**、**架构中心目录**、**架构总入口**、**架构索引**、**所有 feature 的方案 doc**),具体目录位置见主技能 `easysdd` 第二节"目录安排"——那是整个 easysdd 家族目录约定的唯一定义处。

> `{feature}` 是占位符,代表具体 feature 的**目录名**,格式是 `YYYY-MM-DD-{英文 slug}`(日期前缀用于排序,小写连字符 slug;详见主技能 `easysdd` 第二节"目录安排")。到本阶段,feature 目录已由 brainstorm 或 PRD 阶段创建好,这里只消费不新建。

## 你的职责

为这个功能起草一份方案 doc,**逐节让用户拍板**。你的输出节奏不是"一次甩一份完整 doc",而是**写一节 → 让用户确认 → 再写下一节**。这样用户能在便宜的阶段及早叫停跑偏的方向。

## 启动检查

启动时必须先确认下列前置:

1. **PRD 是否存在**。检查 PRD 文档是否存在,内容是否完整(5 节都有内容)。
   - 不存在 → 停下来,告诉用户先走 easysdd-feature-prd 技能,不要替他补 PRD。
2. **是否已有同名方案 doc**。检查方案 doc 是否已存在,有重名先确认是接着改还是新建。
3. **读 PRD 的前置文档**。在动笔前先读:
   - `AGENTS.md`
   - 架构总入口(项目级架构权威)
   - 架构索引(项目级架构 doc 索引)
   - PRD 文档本身
   - PRD 文档中提到的相关架构中心目录下的子系统架构 doc

> 注意区分两类 doc:方案 doc 是**本功能的方案 doc**(本阶段的产出,与同目录下的 `prd.md` / `acceptance.md` 聚合在 feature 目录下);架构中心目录下的 doc 是**项目级架构文档**(AGENTS.md 钦定的架构权威,本阶段的阅读材料)。两者的关系由方案 doc 第 8 节维护。

不读这几份就动笔,几乎保证术语撞车。

## 强制结构

方案 doc 必须有这 8 节,**按顺序逐节写**,每节写完都让用户确认再写下一节。

### 第 0 节:术语约定(最重要,必须先写)

为新功能引入的每个关键概念定义:

```markdown
## 0. 术语约定

### {术语名}
- **定义**:一句话说死
- **出现层**:数据模型 / 协议 / 前端类型 / UI 文案 / 持久化 (列出实际涉及的)
- **外部对应字段**:如对应 OpenAI/其他 API 的某字段,写出来;否则"无"
```

**防撞车检查**(写完每个术语都必须做):

- 用 Grep 搜索:现有代码 + 架构中心目录(项目级架构 doc)+ 所有 feature 的方案 doc(已有的 feature 方案 doc),看这个名字有没有被既有概念占用。
- 撞了 → 二选一:(a) 改新概念的名字 (b) 确认就是同一东西,复用既有命名,**不在 doc 里重复定义**。
- 把 grep 命中情况简短记录下来,让用户能复核。

可选写一份**禁用词列表**:列出新代码里禁止出现的容易混淆的通用词。

> **为什么这节最重要**:术语不锁死,后续每次修 BUG 概念都会漂移一点。这节是整份 doc 对未来维护成本影响最大的一节。

### 第 1 节:需求摘要

把 PRD 浓缩成 3-5 句话。**不加任何设计内容**,纯摘要。目的是让未来读这份 doc 的人(包括 AI)不需要反复跳回 PRD。

### 第 2 节:对接点梳理

新功能不是凭空出现的,要和现有代码接上。逐条列出:

- **读哪些既有数据**(store / 表 / 接口)
- **写哪些既有数据**(⚠️ 最危险,点名哪条写路径是新增的,是否破坏"单一写路径")
- **订阅哪些既有事件**
- **发射哪些既有事件**(在既有事件流注入新事件)
- **改动哪些既有 UI 区域**
- **完全新建、不 touch 既有代码的部分**(单独列)

每一项都要给**代码指针**:文件路径 + 函数/类型名。写不出指针 = 你还没看够代码。

> 对接点梳理时如果发现"既有架构根本容不下这个需求",**停下来**和用户确认要不要升级成重构方案。本工作流不处理重构。

### 第 3 节:目标模型

**字面写 TypeScript / Rust 伪代码**,不要用自然语言描述"大概有哪些字段"。

- 新增类型定义(完整 `type Foo = { ... }`)
- 新增事件 schema(字段名、类型、可选性全写出)
- 新增接口签名(输入、输出、错误)
- 数据流向图(ASCII 或文字步骤):谁产生数据 → 经过谁处理 → 到达谁

> 规则:能用代码表达的不要用散文。散文留给 AI 的自由度太大。

### 第 4 节:关键交互

- 主要用户流程的事件时序(ASCII 流程图或步骤列表)
- 并发 / 异常 / 边界情况各自的预期行为(超时、取消、空输入、权限不足)
- **被否的替代方案**:列 1-2 个考虑过但没选的方向 + 否掉的理由。这一段防止未来回头争论"为什么不那样做"。

### 第 5 节:改动计划

按"新增 / 调整"两类平铺:

- **新增**:新建的文件、类型、组件、事件、接口,对应路径和职责
- **调整**:既有代码改的地方(来自第 2 节对接点),具体到 文件 + 函数 + 改什么
- **删除**(可选):只在替代了某些临时占位代码 / mock 时才有,没有写"无"

每一项能对应到具体路径和函数名。**写不出来就意味着方案还不具体**。

### 第 6 节:推进顺序

按"可独立验证的粒度"切分步骤:

- 每步前置依赖、改动范围、退出信号(怎么算这步做完)
- 相邻两步之间插入人工 checkpoint(阶段四会用到)
- **步骤总数 6-8 步是上限**——超过意味着粒度太细或方案太大,考虑拆子需求

> 推进顺序不是代码行号顺序,是**功能可见度顺序**——先让最小闭环跑通,再叠细节,这样每步用户都能实际验证。

### 第 7 节:不变量清单

留个空标题,内容由阶段三(easysdd-feature-test-spec)回填。

```markdown
## 7. 不变量清单

> 待阶段三填入。
```

### 第 8 节:与项目级架构文档的关系

- 本方案引用了哪些架构中心目录下的 doc(架构总入口、架构索引、子系统架构 doc)?
- 本方案是否需要在项目级架构 doc 里补一段或改一段?**需要的话必须同步改**——AGENTS.md 明确要求项目级架构讨论的落点是架构中心目录,不要在代码里另起炉灶,也不要只写在方案 doc 里就算完
- 注意:本节讨论的是本功能和**项目级架构文档**的关系,不是和其他 feature 方案 doc 的关系。其他 feature 的方案 doc(所有 feature 的方案 doc 中的某一份)的引用,在第 2 节"对接点梳理"里作为读写源给出即可

## 逐节确认节奏

每写完一节,跟用户做一次确认对话,问类似:

- "第 0 节我列了 X / Y / Z 三个新术语,grep 没撞车,你看有没有漏掉你心里想着但 doc 没提的概念?"
- "第 2 节对接点我列了这些既有模块,有没有遗漏?(漏掉的对接点是事后冲突的主要来源)"
- "第 3 节的伪代码字段够不够具体?有没有'你以为我会懂'的地方?"

用户拍板后再写下一节。**不要一口气写完 8 节再让用户 review**——那会浪费大量返工时间。

## 退出条件

8 节都有内容,用户逐节确认,且:

- [ ] 第 0 节做过 grep 防撞车检查
- [ ] 第 2 节每个对接点都有代码指针
- [ ] 第 3 节用伪代码,不是散文
- [ ] 第 5 节"不做什么"和 PRD 第 4 节一致(没有偷偷扩范围)
- [ ] 第 6 节推进顺序 ≤ 8 步,每步都能独立验证
- [ ] 第 7 节占位存在(等阶段三填)

文件路径:方案 doc(feature 目录不存在就创建——同一 feature 的 prd.md / design.md / acceptance.md 都聚合在这个目录下;目录位置见主技能 `easysdd` 第二节"目录安排")

## 退出后

告诉用户:"方案 doc 已就绪,下一步是阶段三:测试设计(列不变量)。可以触发 easysdd-feature-test-spec 技能。"

不要自己顺手开始列不变量——人工 checkpoint 是硬约束。

## 反模式(看到就停)

- ❌ 不读 AGENTS.md / 既有架构 doc 就动笔
- ❌ 第 0 节的术语没做 grep 防撞车
- ❌ 第 3 节用"包含若干字段表示用户信息"这种散文代替伪代码
- ❌ 第 6 节切出 12 步——粒度太细,该合并或拆子需求
- ❌ 一口气把 8 节都写完才让用户 review
- ❌ 第 5 节"不做什么"比 PRD 多了几条"我觉得也该做的"——范围蔓延
