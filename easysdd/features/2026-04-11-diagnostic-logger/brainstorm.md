# Diagnostic Logger Brainstorm

> 阶段: Stage 0 (可选前置·brainstorm)
> 状态: 已确认
> 创建日期: 2026-04-11
> 下一步: Stage 1 PRD

## 1. 问题框架

本次要解决的真问题不是“给项目随便加点日志”，而是为 March 建立一条独立、可分级、可长期回看的诊断日志通道。它需要把运行时关键事件稳定写进 `.march`，在定位问题时可直接回看过程，同时避免和现有 runtime/debug 输出混在一起，导致真正用于排障的日志被海量运行时信息淹没。

对话里被否掉的其他框架：

- “继续扩展现有 `.march/debug/*.log` 就够了”
  这个方向被否掉，因为用户明确希望诊断日志与运行时输出位置分离，避免高频运行时日志掩盖 debug 时真正要看的记录。
- “做一套自动收集上下文的重型日志系统”
  这个方向被否掉，因为用户明确不希望自动抓全量上下文，不希望把日志系统做成默认采集大量快照/对象的重工具。

## 2. 产品压力测试结论

- **不做会怎样**: 出问题时仍然只能在 runtime 输出或零散 debug 信息里翻找，排障过程依赖临时加打印，难以稳定回看关键过程。
- **更高价值的邻近问题**: 有，长期看可能会延伸到日志查看 UI、统一排障工作流、甚至更完整的调试面板；但本次不收，先把日志骨架与分流结构立住。
- **更小版本**: 有。第一版采用“做更少抓 80% 价值”的方式：后端只接 3 到 5 个关键模块，前端只提供手动调试 logger，不做默认全局埋点。
- **时机判断**: 是当前值得做的事。仓库已经存在 `.march/debug/context.log` 与 `provider.log` 一类 debug 输出，说明问题定位需求真实存在；现在补齐独立诊断通道，可以避免后续各模块各自落盘、结构失控。

## 3. 考虑过的方向

### 方向 A: 双通道日志骨架

- 描述: 后端提供正式诊断 logger，按等级写入 `.march` 下独立于 runtime/debug 输出的位置；前端提供单独的轻量 logger，只在调试时显式使用。
- 核心价值: 先把目录、文件分流、等级语义和调用方式定清，后续扩展不会乱。
- 主要代价 / 风险: 第一版覆盖面有限，排障收益主要来自那 3 到 5 个后端关键模块。
- 结论: **选定**。它同时满足“默认克制”和“结构先行”，并且允许前后端分别落地、避免被强行绑成同一实现。

### 方向 B: 后端优先、前端预留

- 描述: 先把后端诊断日志做完整并接入关键模块；前端这次只定义接口和目录约定，真正需要时再补前端日志落盘。
- 核心价值: 最快拿到后端排障收益，复杂度最低。
- 主要代价 / 风险: 前端短期仍没有现成可用的独立调试日志能力。
- 结论: **否决**。用户明确希望前端也有独立 logger，只是不用默认接模块。

### 方向 C: 分级日志规范优先

- 描述: 先重点定义什么级别该进什么文件、什么事件值得打、runtime/debug/diagnostic 三类输出怎么区分，再在后端接少量点位验证；前端只提供手动可调用的 debug logger。
- 核心价值: 最能防止“日志太多”和“接入太乱”，规范先于铺开。
- 主要代价 / 风险: 第一版更偏地基建设，短期可见收益没方向 A 那么直接。
- 结论: **部分吸收**。其“结构先行、等级先行”的思想被纳入选定方向，但不单独作为最终方案。

## 4. 选定方向

选定方向是“`.march` 下的双通道日志骨架”。后端有一套正式的诊断日志能力，默认接入 3 到 5 个关键模块并按日志等级输出到独立位置；前端有单独的轻量 logger，仅在调试时显式调用，不做默认全局埋点。两条通道都与现有 runtime/debug 输出分开，避免排障日志被噪音淹没。

## 5. 粗粒度边界 (供 PRD 参考，不是最终范围)

- **核心用户感知行为**: 后端关键链路默认有诊断记录，前端调试时也能手动落盘，而且两边都不会和 runtime 输出混在一起。
- **明确不做** (候选):
  - 不做自动抓全量上下文，不自动把大对象、完整快照、整段输出全部写入日志。
  - 不做前端默认全局埋点；前端 logger 第一版是调试时手动使用的工具。
  - 不做日志查看 UI；第一版只保证 `.march` 落盘、目录结构和输出规则清楚。
- **最大未知 / 风险**: 日志等级和文件分流语义是否足够稳定，例如不同等级该去哪里、runtime/debug/diagnostic 三类输出的边界如何避免混淆。
- **成功的粗粒度描述**: 调试前端问题时，开发者可以手动打开一条独立日志，不会和后端/运行时日志混在一起；同时后端关键链路已有基础诊断记录可供排查。

## 6. 遗留给 PRD 阶段的问题

- 诊断日志在 `.march` 下的目录结构、命名约定和文件分流规则具体是什么。
- 日志等级的最小集合是什么，以及每个等级分别适合记录哪些事件。
- 现有 `.march/debug/*.log` 与新诊断日志之间的职责边界如何定义。
- 后端第一版优先接入的 3 到 5 个关键模块具体是哪些。
- 前端轻量 logger 的启用方式和最小 API 形态是什么。
- 是否需要最基础的轮转/清理策略，避免 `.march` 日志目录长期膨胀。

## 备注

- 仓库扫描发现现有 CLI 调试能力已把内容写入 `.march/debug/context.log` 和 `.march/debug/provider.log`，说明日志落盘并非全新概念，本次重点是补齐独立诊断通道而不是从零引入“写文件日志”。
- 本次 brainstorm 未进入技术实现层，不讨论具体 Rust/Vue logging 库、存储格式或 UI 呈现方式。
