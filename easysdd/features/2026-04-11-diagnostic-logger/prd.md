# Diagnostic Logger PRD

> 阶段: 阶段一 (PRD)
> 状态: 草稿
> 创建日期: 2026-04-11

## 1. 问题与场景

当 March 的开发者在调试后端关键链路异常、运行时状态变化异常，或前端交互问题时，现在只能翻 runtime 输出或临时加打印，难以稳定回看关键过程；所以需要一套写入 `.march` 独立位置、支持日志等级、且不与现有 runtime/debug 输出混在一起的诊断日志能力。

## 2. 用户可感知行为

开发者在排查问题时，可以查看一份独立于 runtime 输出的诊断记录；后端关键链路默认会留下可回看的事件记录，前端在需要调试时也可以主动留下独立记录，因此排障时不需要再先从混杂的运行时输出里筛信息。

## 3. 成功长什么样

当开发者调试前端问题时，可以手动留下独立日志且不会与后端或 runtime 输出混在一起；当开发者排查后端关键链路问题时，无需临时加打印，就能直接回看关键事件记录并定位问题经过。

## 4. 不做什么

- 不做自动抓取全量上下文、完整快照或大对象日志。
- 不做前端默认全局埋点，只支持调试时主动记录。
- 不做日志查看 UI 或可视化面板。
- 不在第一版里覆盖所有后端模块，只接入 3 到 5 个关键模块。

## 5. 硬约束

- 必须落在 `.march` 目录下，但要与现有 runtime/debug 输出分开。
- 前后端不能共用同一个 logger 实现，需要分别设计。
- 第一版必须支持日志等级。
- 第一版后端只接入 3 到 5 个关键模块。
- 其他硬约束：无。

## 备注

- 本次 PRD 基于同目录下的 brainstorm note 收敛而来。
- 当前仓库已经存在 `.march/debug/context.log` 和 `.march/debug/provider.log` 一类调试输出；本功能的目标不是替代它们，而是补一条独立的诊断日志通道。
