# UI 设计：任务自动命名

> 本文定义 Ma UI 中 task 的“延迟命名”机制。它是 [`DESIGN.md`](DESIGN.md) 中“一个主题就是一个对话框”在任务创建体验上的延伸：用户先进入主题窗口开始对话，标题再根据真实主题自动生成，而不是在进入前被迫先起名。

---

## 目标

解决当前“新建任务前必须先填写名字”的阻塞感，同时避免任务列表里长期堆积大量 `新任务`、`默认任务` 这类弱语义标题。

设计目标：

- 用户点击“新建任务”后应立刻进入可对话状态，而不是先被弹窗打断
- task 标题应尽量反映用户第一次真实提问的主题
- 自动命名不能阻塞主对话链路，不能影响第一轮正常执行
- 用户一旦手动改名，系统后续不应再偷偷覆盖

---

## 核心原则

### 1. 先创建主题窗口，再命名

task 的本体是“一个持续演进的主题窗口”，不是“一个必须预先命名的表单对象”。

因此：

- 点击“新建任务”时，先创建一个临时标题 task
- 焦点直接进入聊天输入框
- 当用户发出第一条消息后，再基于这条消息生成标题

### 2. 自动命名只发生在首轮

自动命名只针对“尚未正式命名”的 task：

- task 当前标题仍是系统默认标题
- task 尚无历史对话
- 用户尚未手动改名

一旦 task 已被自动命名，或已被用户手改，系统不再尝试根据后续消息改名。

### 3. 标题生成与主 agent 解耦

标题生成不应依附在主 agent 回复里“顺带完成”，而应走一条独立的轻量命名链路。

原因：

- 标题和回复的优化目标不同
- 标题需要短、稳、可扫描
- 单独命名更容易约束输出，也更容易做失败降级

---

## 交互流程

```text
用户点击“新建任务”
  -> 立即创建 task（标题先为“新任务”）
  -> 左栏出现新 task，并切到该 task
  -> 输入框自动聚焦

用户发送第一条消息
  -> 后端正常执行这一轮 agent
  -> 同时判断该 task 是否满足“首轮自动命名”条件
  -> 若满足，则触发一次标题生成
  -> 成功后写回 task.name，并刷新左栏
```

---

## 数据模型

为避免未来出现“AI 自动命名覆盖用户手改标题”的问题，task 持久化需要显式记录标题来源与锁定状态。

### `tasks` 表新增字段

```sql
ALTER TABLE tasks ADD COLUMN title_source TEXT NOT NULL DEFAULT 'default';
ALTER TABLE tasks ADD COLUMN title_locked INTEGER NOT NULL DEFAULT 0;
```

含义：

- `title_source`
  - `default`：系统默认标题，尚未正式命名
  - `auto`：由自动命名生成
  - `manual`：由用户手动输入或手动修改
- `title_locked`
  - `0`：允许系统在“默认标题 + 首轮”场景下自动命名
  - `1`：标题已被用户确认或手改，系统不可再自动覆盖

当前 MVP 中虽然尚未提供 task 重命名 UI，但数据层先保留这两个字段，避免后续补重命名时再返工数据库语义。

---

## 命名规则

输入：

- 仅使用该 task 的第一条用户消息
- 可选附带极少量稳定上下文，例如工作目录名

输出要求：

- 以 8-18 个字符为优先区间
- 尽量使用动宾结构或问题定位结构
- 不保留“帮我”“请你”“看一下”这类口语前缀
- 不输出引号、句号、冒号等结尾装饰
- 尽量保留关键对象，例如文件名、模块名、问题对象

例子：

- “帮我看看 main.rs 这里有没有问题”
  -> `检查 main.rs 问题`
- “把登录模块拆一下，顺便补测试”
  -> `重构登录模块并补测试`
- “这块为什么会重复渲染？”
  -> `排查重复渲染问题`

---

## 运行时规则

当用户发送消息时，系统按以下条件判断是否自动命名：

1. 该 task 的 `title_source == "default"`
2. `title_locked == false`
3. 当前消息是该 task 的第一条用户消息

若满足：

- 主 agent 正常继续执行
- 标题生成作为附加步骤执行
- 若命名成功，则将 task 更新为：
  - `name = generated_title`
  - `title_source = "auto"`
  - `title_locked = false`

若失败：

- 保持原默认标题
- 不影响这一轮消息与回复

---

## 降级策略

自动命名不应成为主链路依赖，因此必须允许静默失败并降级：

- provider 不可用：保留默认标题
- 输出为空或过长：走本地规则裁剪
- 首条消息过短（如“继续”“看看这个”）：保留默认标题
- task 在命名过程中已被删除：放弃写回

本地规则降级可基于第一条消息做简单清洗与截断，作为 provider 命名失败时的 fallback。

---

## 前端策略

MVP 前端交互改为：

- 删除“新建任务”命名弹窗
- 点击“新建任务”后直接创建一个默认标题 task
- 创建成功后自动聚焦聊天输入框

task 名称编辑能力不再发生在创建前，而应作为后续 task 操作的一部分补充进来。

---

## 与现有设计的关系

这套机制延续了 [`DESIGN.md`](DESIGN.md) 中的两个核心理念：

- **一个主题就是一个对话框**：先进入主题开始工作，再由系统总结标题
- **用户视图 vs AI 上下文分离**：标题是用户侧组织信息，不进入 AI 上下文核心结构，不应干扰首轮上下文构建
