## 背景

March 最近连续做了几轮 UI 减法和密度调整。过程中为了收左栏、右栏、聊天区、输入区的噪音，局部出现了 `9px / 10px / 11px / 12px / 13px` 混用的情况。

这种“每块顺手调一点”的方式短期能解决单点问题，但会带来两个长期问题：

1. 同一层级的元素在不同面板里字号不一致，界面看起来像几个独立系统拼在一起
2. 后续继续调样式时，AI 和人都缺少统一基准，只能继续凭截图和感觉微调

March 的目标不是做一套花哨的视觉稿，而是做一个“永不腐烂的工作台”。这要求字号系统本身也足够克制、稳定、可追溯。

## 决定

March 的持久 UI 字号统一采用一套四档主尺度：

- `12px`：caption / meta / 次级状态
- `13px`：普通 UI 文本与交互控件
- `14px`：主阅读文本
- `16px`：区域标题

同时新增一条例外约束：

- `10px` 只保留给高密度且需要主动退后的内容，例如 mono 调试信息、trace、token 用量、极短状态标签

不再把 `9px` 和 `11px` 作为常规持久 UI 档位使用。它们若继续存在，应视为待收敛的过渡状态，而不是长期规范。

为便于执行，样式层同步提供以下 token：

- `--ma-font-size-dense: 10px`
- `--ma-font-size-caption: 12px`
- `--ma-font-size-ui: 13px`
- `--ma-font-size-body: 14px`
- `--ma-font-size-title: 16px`

## 理由

### 1. 先保秩序，再谈细节

March 当前最突出的问题不是“某个字太大或太小”，而是缺少一套统一的字号阶梯。四档主尺度足以覆盖三栏布局里绝大多数持久信息层级，又不会把系统做得过于复杂。

### 2. 让主次关系更稳定

`12 / 13 / 14 / 16` 的间隔足够明确，能把“脚注级信息”“普通 UI”“主阅读内容”“区域标题”清楚分开，避免出现“正文像脚注、脚注像正文”的情况。

### 3. 符合 March 的减法方向

March 的视觉语言强调工作台气质、克制和长期稳定。减少字号档位，本质上是在拿掉解释性噪音和局部特例，让层级更多依赖一致的节奏，而不是临时补丁。

### 4. 给后续实现一个机械化基准

一旦字号被定成约定，后续做任务列表、Context、聊天区、设置页时，优先问题就从“这里该写几 px”变成“这个元素属于哪一层级”。这会显著减少反复来回的视觉微调。

## 考虑过的替代方案

### 方案 A：继续按区域逐个微调

没有采用。这样虽然局部灵活，但系统会继续漂移，每轮改动都会把“当前截图”变成新的临时标准。

### 方案 B：采用更细的完整字体刻度（如 10 / 11 / 12 / 13 / 14 / 16 / 18）

没有采用。March 当前的主要问题不是档位不够多，而是档位太容易被随手使用。先把主尺度收敛到四档，更符合当前阶段的减法目标。

### 方案 C：强行禁止 10px

没有采用。右栏调试、token、trace、mono 状态这类信息确实需要更退后的极小字号；完全禁止会导致这些内容和主界面抢同一层注意力。

## 后果

1. 以后新增或调整持久 UI 时，先判断它属于 `caption / ui / body / title` 哪一层，再选字号，而不是直接写任意 `text-[Npx]`
2. 现有样式中仍然存在的 `9px` 和 `11px` 应视为技术债，后续触达相关区域时优先往四档主尺度收敛
3. 调试、trace、token、mono 紧凑信息可以继续使用 `10px`，但必须证明它们确实应该退后，不可扩散到普通内容区
4. 若未来要引入更系统的 typography token 或 utility class，应以这套四档主尺度为基础，而不是重新发明一套并行体系
