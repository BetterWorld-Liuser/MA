# Ma — 设计文档

> **Ma**（码/马）— 2026 马年诞生，码农的伙伴，极简有力。也可以叫 **M A**，两个字母，分开念。命令行直接 `ma "帮我重构这个函数"`。

## 项目目标

构建一个以**上下文管理**为核心竞争力的 agentic coding 工具，用 Rust 实现。Ma 的核心差异化：AI 永远感知代码的真实状态，不被过时的上下文迷惑。

---

## 核心理念

### 问题：现有工具的上下文缺陷

现有工具（OpenCode、Claude Code 等）在文件编辑后，上下文保留的是：
```
[read_file 返回的原始内容]
[str_replace 的 old/new 片段]
[tool_result: success]
```
原文件内容永远留在上下文里，即使文件已经被改掉。改 10 个文件 = 10 份原始内容堆在上下文中。

### 解决方案：文件系统作为 Source of Truth

- 用 **file watcher** 监控工作目录
- 无论是 AI 改的、用户手动改的、还是其他程序改的，上下文里的文件内容**永远反映磁盘真实状态**
- AI 看到的不是"我上次写进去的内容"，而是"文件现在实际长什么样"

## 架构设计

### 双轨设计：用户视图 vs AI 上下文分离

```
用户看到的                    AI 收到的
─────────────────            ─────────────────
完整聊天记录                  精简后的上下文
所有历史对话                  只有近几轮对话
工具调用摘要                  当前文件真实快照
```

用户永远看到完整对话，AI 收到的是精简过的上下文。

## 工具哲学

AI 以命令执行为主，但不能只有命令行；基础文件读写能力应作为一级工具存在，文件修改仍通过行号精确定位。

- `run_command`：负责调用编译、测试、git、grep、脚本等环境能力
  并且允许显式指定命令执行环境，例如 `bash` / `powershell` / `cmd`
  Ma 会先扫描当前环境中真实可用的 shell，并把可选项注入给 AI
- `read_file` / `write_file`：负责最基础、最稳定的文件读取与落盘
- 行号级编辑能力：负责精确修改文件片段，避免基于文本匹配的脆弱替换

这样设计的原因不是回到“高层魔法工具”，而是明确区分两类能力：

- 命令执行是和外部环境交互
- 文件工具是和 Ma 自己维护的 Source of Truth 直接交互

前者保留通用性，后者保证基础文件操作简单、稳定、可控。
