# 聊天区与输入框设计

> 从 [ui.md](ui.md) 延伸：聊天区承载完整用户视图，输入框负责用户与 March 的交互入口。

---

## 聊天区定位

聊天区负责承载用户看到的完整对话历史，同时把当前轮 agent 的推进过程可感知地呈现出来。

目标不是把中栏做成日志终端，而是让用户始终理解三件事：

- 我刚刚发出的请求已经被接收
- AI 当前这轮正在推进，而不是卡住
- 当前轮具体做了哪些关键动作

---

## 基础结构

`Task.timeline` 是 UserMessage 和 Turn 按 `seq` 交错的扁平列表，`<ChatMessageList />` 按序遍历渲染：

```text
┌──────────────────────────────────────────────────┐
│  ← UserMessage ──────────────────────────────── │
│                              用户  14:32          │
│              帮我把 auth 模块拆成更小的单元         │
│                                                  │
│  ← TurnGroup（Turn state: done）──────────────── │
│  🤖 March  ← 来自用户消息                         │
│  ▶ 3 个动作                                      │  ← 折叠摘要（点击展开）
│  ─────────────── 最终消息 ─────────────────────── │
│  已完成，auth 模块拆成了三个文件                    │
│                                                  │
│  ← UserMessage ──────────────────────────────── │
│                              用户  14:35          │
│   @reviewer @architect 分别看看拆分结果            │
│                                                  │
│  ← TurnGroup（Turn state: streaming）──────────  │
│  🔵 代码审查员  ← 来自用户消息                      │
│  ┌─ read_file src/auth_token.rs                 │
│  ├─ read_file src/auth_session.rs               │
│  └─ ...                                         │
│                                                  │
│  ← TurnGroup（Turn state: streaming）──────────  │
│  🟢 架构师  ← 来自用户消息                         │
│  ┌─ read_file src/auth_token.rs    ✓            │
│  └─ ...                                         │
├──────────────────────────────────────────────────┤
│ 输入框                              [中断所有][发送] │
└──────────────────────────────────────────────────┘
```

- 完整对话历史，用户可以翻回去
- UserMessage 固定在右侧；TurnGroup 固定在左侧，头部显示 agent 名 + 触发来源
- 并发 streaming 的多个 Turn 就是相邻的独立卡片，不做 side-by-side 或嵌套缩进
- 顶部不放置无信息增益的重复标题

## 消息状态模型

中栏数据结构是 `TaskTimelineEntry[]`——按 `seq` 严格排序的扁平列表，条目只有两种：`UserMessage` 和 `Turn`。

**Turn 的四种状态：**

| 状态 | 含义 |
|------|------|
| `streaming` | 该 turn 正在执行（可能有多个并发 turn 同时处于此状态） |
| `done` | 该 turn 正常结束，已折叠为「N 个动作 ▸」摘要 + 最终消息 |
| `failed` | 该 turn 出错，折叠摘要后显示错误信息，提供重试入口（重试 = 重新发送同一条用户消息） |
| `cancelled` | 用户主动取消，已产出内容保留并折叠，带轻量视觉标记 |

**并发执行**：一条 UserMessage 可通过 `mentions`（@Agent）和 `replies`（引用 Turn）同时激活多个 agent，各自独立运行为独立的 Turn。任意时刻同一 task 内可能存在多个 `state='streaming'` 的 Turn，它们在时间线上是相邻的独立卡片，各自 streaming、各自独立折叠，互不干扰。

用户发送消息时，前端立即在时间线末尾乐观插入一条 UserMessage（临时 id）。收到 `user_message_appended` 后替换为正式 id，随即按激活列表依次收到若干 `turn_started`，每个 turn 在时间线末尾插入对应的空 Turn 卡片。

### Turn 内展示策略

一个 Turn 内部含 1+ 条 Message（每条对应一次 LLM API 调用）。Turn 处于 `streaming` 状态时，所有 Message 的 reasoning 与 timeline（TextChunk 和 ToolCall 交错）**按事件到达顺序完全平铺**，呈现为一条连续的操作流水，不做任何预测性分割。

Turn 收到 `turn_finished` 时，前端执行后验折叠：

1. 该 turn 最后一条 Message 为最终 Message
2. 最终 Message 之前的所有内容折叠为「N 个动作 ▸」摘要（N 只统计 ToolCall 个数）
3. 摘要与最终 Message 之间画 `── 最终消息 ──` 分割线
4. 最终 Message 保持展开；点击摘要可展开恢复平铺

**边界情况**：Turn 内只有一条 Message 时，不画摘要、不画分割线，直接展开该 Message。

### 折叠原子性

`turn_finished` 是某个 Turn 折叠的**唯一触发时机**：

- 不允许在 `message_finished` 时触发折叠或分割线渲染
- 不允许 snapshot 刷新连带改变 streaming Turn 的视图状态
- 并发 streaming 的多个 Turn 各自独立在收到自己的 `turn_finished` 时折叠，互不影响
- 右栏 debug trace / context usage 的更新不得导致聊天时间线重建或已完成 Turn 的折叠状态重置

### 中栏数据源边界

中栏消息流必须有自己独立的数据源，不允许作为 `active_task snapshot` 的派生视图存在。

具体约束：

- 中栏只读取 `chatRuntimeStore`，形态为 `Record<taskId, TaskTimelineEntry[]>`
- `TaskTimelineEntry` 是 `UserMessage | Turn` 的判别联合，是中栏**唯一 source of truth**，不存在 liveTurn 或独立历史消息列表的双结构
- task snapshot 只能在以下场景参与聊天区：
  - 首次打开某个 task（`get_task_history` 拉基线）
  - 切换到另一个 task
  - 应用重启后的恢复
  - `gap_too_large` 触发重拉

不允许的路径：

- 右栏 `notes / open_files / debug_trace / context_usage` 更新时顺带重建聊天列表
- `send_message` 返回后再用整段 `history` 回填中栏
- 聊天区同时消费事件 append 和 snapshot hydrate 两条来源（snapshot 只在切换/初始化时写入）

设计目标：

- 用户发送后，UserMessage 立即乐观插入 timeline（临时 id），收到 `user_message_appended` 后替换正式 id
- 运行过程中，所有内容通过事件流的 `chatEventReducer` 增量写入对应 Turn 的 messages
- `turn_finished` 到达时，目标 Turn state 置为 done，仅此一次触发折叠，无额外写入动作

## 等待态原则

等待态必须建立在真实事件之上，不伪造"AI 正在努力思考"的表演。

**应展示**：简短状态文案（`正在整理上下文`、`正在读取文件`、`正在执行命令`）、轻量动态提示（点状脉冲、弱 shimmer）、当前轮已发生的工具摘要。

**不应展示**：无含义的假进度条、没有事件却持续切换的"思考文案"、把底层细节强行暴露成调试日志。

区分两类等待来源：

| 来源 | 展示方式 |
|------|--------|
| AI 思考中（未产生首 token） | 轻量抽象动态提示 |
| 工具执行中 | 动作名称 + 对象 + 耗时 |

---

## 工具调用展示

- 新工具开始时，立刻在当前 AI 消息下插入一条工具项（实时，不等整轮结束）
- 默认显示紧凑摘要，如 `read_file src/auth.rs`、`run_command cargo test`
- 展开后显示原始参数和输出
- 工具项状态：`running`（spinner + 耗时）→ `success` / `error`

---

## 与右栏联动

- 工具读取文件时，右栏对应监控文件短暂高亮
- 文件写入成功后，右栏新旧感立即刷新
- 新一轮上下文构建完成后，右栏上下文用量同步更新
- 这些联动不应破坏聊天区自己的局部状态，例如滚动位置、live turn 过渡、已沉淀消息列表

设计约束：

- 中栏消息流和右栏 debug/context 面板共享运行事实，但不共享“整块刷新节奏”
- 右栏状态更新默认应视为旁路更新，不干扰聊天区动画与消息布局

## 实现提示

围绕"轮次事件流"建模，而不是让前端靠零散布尔值拼状态。事件类型与状态边界草案详见 [ui-events.md](ui-events.md)。

## 输入框

### 布局

```
┌──────────────────────────────────────────────────────────────┐
│  帮我重构 [auth.rs ×] 的认证逻辑…                             │
│                                                              │
├──────────────────────────────────────────────────────────────┤
│  [+]  [📁 apps/web ∨] [⚡ claude-sonnet-4-6 ∨] [💡 思考 8K ∨] [↵发送] │
└──────────────────────────────────────────────────────────────┘
```

- 输入区自动扩展高度，最高约 6 行后出现内部滚动
- `@` 引用的文件以 chip 形式内联显示，chip 上有 `×` 可移除
- `Enter` 发送，`Shift+Enter` 换行
- 发送后立即清空输入框；当前轮运行时允许用户继续在输入框中编辑下一条消息，也允许滚动历史；若停留在底部则自动跟随流式内容
- 输入区底栏提供任务级工作目录入口；目录按钮默认显示当前目录名，hover/tooltip 显示绝对路径
- `💡` Reasoning 控件仅当当前模型支持 reasoning 时显示，三种 style 对应不同形态；详见 → [Reasoning 模型支持 · 输入框底栏控件](reasoning.md#输入框底栏reasoning-控件)

### 运行中输入

- 当前 task 有 streaming Turn 时，输入框、`@` 面板、图片附件区和任务级运行上下文控件应继续可用，不因为”AI 正在回复”而整体 disabled
- 运行中底栏主按钮切换为”中断所有”（等效于 `cancel_task`），每个 TurnGroup 卡片各自提供独立的取消按钮（等效于 `cancel_turn`）；不隐藏”发送”按钮，但用户发送新消息会在当前轮仍 streaming 时排入等待——具体策略取决于后端是否支持队列；无论如何，前端不得偷偷排队隐式请求
- 若用户在运行中按 `Enter`，只有当后端确认可接受新消息时才创建新一轮；否则应给出明确提示
- 当所有 streaming Turn 结束后，输入框里保留的草稿就是下一轮候选输入
- 这样设计的目标是把”AI 后台在跑”和”用户前台继续组织下一条请求”解耦，避免把等待时间浪费成纯只读状态

### `@` mention

用户输入 `@` 触发 mention 面板，浮于输入框上方：

```
┌──────────────────────────────────────────────┐
│  > rev                                       │  ← fuzzy 搜索
│  👤 reviewer 代码审查员                       │  ← 角色
│     重点审查实现风险、边界条件、测试缺口…      │
│  src/review.rs                               │  ← 文件
│  src/review/                                 │  ← 目录
└──────────────────────────────────────────────┘
```

三类目标，行为不同：

| 类型 | 行为 |
|------|------|
| `@Agent` | 进入 UserMessage 的 `mentions` 列表，驱动后端为该 agent 创建独立 Turn；在文本中以高亮 tag 内联显示；角色结果在面板中排在文件前面 |
| `@文本文件` | 加入 `open_files`，出现在右栏监控区 |
| `@图片文件` | 转为 image content block 注入本条消息，不加入 `open_files`（需模型支持 vision） |
| `@目录` | 注入目录树结构到本条消息，不加入 `open_files` |

Agent 项除 `name` 和 `display_name` 外，还应展示一条简短描述，帮助用户快速判断“这个角色适不适合接这件事”。这一行描述也应与 prompt 中的 agent roster 使用同一份数据，避免 UI 和 AI 对角色职责的理解漂移。

图片文件的判断基于扩展名（`.png`、`.jpg`、`.jpeg`、`.gif`、`.webp`、`.bmp`、`.svg`），不做 MIME 嗅探。

**目录树注入规则**：遵守 `.gitignore` 过滤；超过 2,000 行截断，附注 `[目录树共 X 项，已截断至 2000 行]`；只显示路径结构，不展开文件内容。

### `/` Skills

用户输入 `/` 触发 skill 面板，浮于输入框上方：

```text
┌──────────────────────────────────────────────┐
│  > rust                                      │
│  ✦ rust                                      │
│    Rust 项目工作流                           │
│    C:/Users/alice/.agent/skills/rust/SKILL.md│
└──────────────────────────────────────────────┘
```

行为规则：

| 类型 | 行为 |
|------|------|
| `/Skill` | 选中后把对应 `SKILL.md` 加入本条消息的待注入上下文；真正发送时先写入 `open_files`，因此后续轮次都按普通文件追踪 |

- `/` 面板只展示当前 task 已发现的可用 skills，不混入文件、目录或角色
- 选中后以 chip 形式显示在输入框中，和 `@文件` 一样可移除
- 发送前前端先调用 `open_files` 打开这些 `SKILL.md`，再发起 `send_message`
- 设计目标不是“把 skill 文本偷偷拼进消息”，而是把 skill 作为用户显性放上桌面的文件进入 Source of Truth

### 模型选择器

```
[⚡ claude-sonnet-4-6 ∨]
   ↓ 点击展开
   ┌──────────────────────────┐
   │ ✓ claude-sonnet-4-6      │
   │   claude-opus-4-6        │
   │   gemini-2.5-pro         │
   │   ──────────────────     │
   │   管理 Provider...       │
   └──────────────────────────┘
```

- 列表来自已配置的 provider，按 provider 分组
- 收起状态优先显示当前已选模型；若当前 task 还未显式选择，则显示该 provider 的当前默认模型；都没有时才显示 `选择模型`
- 展开时优先显示该 provider 最近一次成功读取到的本地缓存列表，再在后台刷新；如果 provider 返回的新列表有变化，再无闪烁地更新
- 每个 task 独立记录选择；默认运行配置只在新建 task 时提供初始值
- 模型切换后，March 重新解析 `ModelCapabilities`，图片入口、工具集等立即跟随变化

模型按钮旁还应提供一个轻量“运行参数”入口，用于调整 task 级模型参数（如 `temperature`、`max_output_tokens`）。它和模型选择一样属于当前 task 的运行上下文，不写回 provider 的模型能力表。

模型能力的数据模型、解析优先级、设置页 UI 详见 → [Provider 设计 · 模型能力解析](provider.md#模型能力解析)

### 工作目录选择器

`working_directory` 是 task 级运行上下文，不是应用全局设置。

```
[📁 apps/web]
   ↓ 点击
   系统目录选择器
```

- 点击后打开系统目录选择器，选中的目录立刻持久化到当前 task
- 若当前 task 使用的是默认目录（workspace 根），按钮显示根目录名且不强调“自定义”
- 若当前 task 改成了其他目录，按钮应给出轻微激活态，并提供“恢复默认工作目录”的快捷入口
- 切换 task 时，该按钮显示的目录、`@` 搜索范围、运行时 `cwd`、`AGENTS.md` 自动接入与 Skills 自动发现都一起切换

### 图片输入

详细设计见 → [图片输入](images.md)。以下是聊天区 UI 层面的行为。

**输入方式**：

- **粘贴**：`Ctrl+V` 粘贴剪贴板中的图片
- **拖入**：拖拽图片文件到输入框
- **`+` 菜单**：选择图片文件
- **`@` 引用**：`@screenshot.png`，March 检测到是图片文件后走图片通道

**输入框预览**：

```
┌─────────────────────────────────────────────────────┐
│  这个页面的布局有问题，帮我看看                         │
│  ┌────────┐ ┌────────┐                              │
│  │ 🖼 ×   │ │ 🖼 ×   │  ← 图片缩略图，× 可移除       │
│  └────────┘ └────────┘                              │
├─────────────────────────────────────────────────────┤
│  [+]  [📁 apps/web ∨] [⚡ claude-sonnet-4-6 ∨] [↵发送] │
└─────────────────────────────────────────────────────┘
```

- 图片以小缩略图形式显示在输入文本下方，每张带 `×` 可移除
- 缩略图点击可放大预览
- 发送后图片随消息一起出现在聊天历史中

**模型不支持时的行为**：

- `+` 菜单中"选择图片"项不显示
- 粘贴/拖入图片时，输入框下方短暂提示："当前模型不支持图片输入"
- `@` 引用图片文件时，同样提示并拒绝

## Skills 在右侧面板的展示

Skills 显示在右栏底部：

```
[Skills]
  rust         (已打开)
  git
  api-style
```

- `(已打开)` 表示 AI 已 `open_file` 读取该 skill
- `(已打开)` 也包括用户通过 `/` 显性选择、并在发送前加入 `open_files` 的 skill
- 右栏本身不承担打开动作；显性调用入口在输入框 `/` 面板
