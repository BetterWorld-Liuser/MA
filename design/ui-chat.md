# UI 设计：聊天区与运行反馈

> 本文定义 Ma 中栏聊天区的消息流、输入交互、等待态与工具调用展示。整体布局与右栏面板详见 [ui-shell.md](ui-shell.md)。

---

## 聊天区定位

聊天区负责承载用户看到的完整对话历史，同时把当前轮 agent 的推进过程可感知地呈现出来。

目标不是把中栏做成日志终端，而是让用户始终理解三件事：

- 我刚刚发出的请求已经被接收
- AI 当前这轮正在推进，而不是卡住
- 当前轮具体做了哪些关键动作

---

## 基础结构

```text
┌──────────────────────────────────────────────┐
│ Ma  14:32                                    │
│ 好的，我先看一下现有结构……                     │
│ ┌─ read_file src/auth.rs                    │
│ ├─ replace_lines 12-30                      │
│ └─ reply ✓                                  │
│                                              │
│                           用户  14:32        │
│                     帮我把 auth 模块拆成更小的单元 │
│                                              │
│ Ma  14:33                                    │
│ 已完成，auth 模块拆成了三个文件                │
├──────────────────────────────────────────────┤
│ 输入框                                 [发送] │
└──────────────────────────────────────────────┘
```

- 完整对话历史，用户可以翻回去
- AI 消息固定在左侧，用户消息固定在右侧
- 整体应表现为“对话流”，而不是一组结构相同的日志卡片
- 顶部避免放置无信息增益的重复标题，例如 `Conversation` / `Chat`
- 聊天区头部只保留必要状态，不重复显示 `ready` 这类从输入区已能感知的状态

---

## 消息状态模型

用户发送消息后，不应让界面进入“静止空白”的等待区间。中栏应该立刻插入一条新的 AI 消息槽位，并让这条消息经历清晰的状态变化：

1. `pending`：用户消息已送出，AI 轮次已创建，但还未开始产出内容
2. `running`：当前轮正在执行，可能在构建上下文、调用工具、等待模型输出
3. `streaming`：AI 正在流式输出自然语言内容
4. `done`：当前轮完成
5. `error`：当前轮出错，显示错误摘要和可重试入口

这个状态模型的价值是：用户第一时间就能看到“系统接单了”，而不是只能盯着输入框等待。

---

## 等待态原则

等待态可以有动态效果，但必须建立在真实事件之上，而不是伪造一种“AI 正在努力思考”的表演。

### 应展示什么

- 简短状态文案：`正在整理上下文`、`正在读取文件`、`正在调用模型`、`正在执行命令`
- 轻量动态提示：点状脉冲、弱 shimmer、细 spinner
- 当前轮已发生的工具摘要
- 与当前轮相关的文件/上下文变化联动

### 不应展示什么

- 长时间无含义的假进度条
- 明明没有事件，却持续切换很多“思考文案”
- 把所有底层细节强行暴露成调试日志

### 设计结论

Ma 的等待态本质上不是一个动画组件，而是**当前轮运行状态的可视化容器**。

## 工具调用展示

工具调用以折叠摘要内联展示，默认收起，点击展开原始 input/output；但展示时机不是等整轮结束后一次性补出，而是随着事件流实时出现。

### 展示方式

- 新工具开始时，立刻在当前 AI 消息下插入一条工具项
- 工具运行中显示进行态和耗时
- 工具完成后切换为成功/失败状态
- 默认显示紧凑摘要，例如 `read_file src/auth.rs`、`run_command cargo test`
- 展开后才显示原始参数和输出

### 原因

这让用户感受到的是“AI 正在一边工作一边汇报关键动作”，而不是最后把一堆日志塞进回复里。

### 工具项状态

- `running`：显示 spinner 和已耗时
- `success`：显示完成标记，可展开详情
- `error`：显示失败标记和错误摘要，必要时在最终回复中解释影响

---

## 思考中 vs 工具执行中

聊天区需要区分两类等待来源，因为它们对用户的含义不同：

### AI 思考中

- 尚未产生自然语言输出
- 可能正在构建上下文或等待模型首 token
- 使用轻量、抽象的动态提示即可

### 工具执行中

- 已知正在执行具体动作
- 应优先展示动作名称、对象和耗时
- 若命中真实文件或命令，摘要应尽量可读

这样用户不会把“模型响应慢”和“命令执行慢”混为一谈。

---

## 与右栏联动

聊天区是主叙事区，右栏是上下文投影区，两者应轻量联动：

- 工具读取文件时，右栏对应监控文件短暂高亮
- 文件写入成功后，右栏更新时间或新旧感立即刷新
- 新一轮上下文构建完成后，右栏上下文用量同步更新

联动的目的是增强可理解性，不是把界面做成到处闪动的监控大盘。

---

## 输入框交互

- 输入框支持 `@文件路径` 触发 open_file
- `Shift+Enter` 换行，`Enter` 发送
- 发送后立即清空或锁定输入框，避免用户误判为未发送成功
- 当前轮运行时仍允许用户滚动历史，不强制跳到底部；但若用户停留在底部，应自动跟随最新流式内容

---

## 实现提示

由于 UI 已约定通过 Tauri event 接收后端推送，建议围绕“轮次事件流”建模，而不是让前端靠零散布尔值拼状态。事件类型、字段与状态边界的草案详见 [ui-events.md](ui-events.md)。

这样聊天区的等待态、工具时间线、右栏联动和最终消息落盘都能共用同一套状态源，避免出现“视觉上很忙，但状态并不真实”的问题。
